## Output Format — STRICT XML, NO MARKDOWN FENCES

### Step 1 — preview the directory tree
Emit exactly one `<tree>` block listing every file you will produce. Indent
children with 2 spaces; use `/` suffix for directories.

```
<tree>
src/
  main/
    App.tsx
    util/
      logger.ts
package.json
README.md
</tree>
```

### Step 2 — emit each file in order
For each file in the tree, emit a `<file>` block. The full content goes
between the opening and closing tags. Do NOT wrap content in markdown
fences.

```
<file path="package.json" op="create">
{full content here}
</file>
```

Valid `op` values:
- `create` — file does not exist; emit full content.
- `write`  — file exists; overwrite full content.
- `skip`   — file exists and is already correct; emit `<file path="..." op="skip" reason="..." />` self-closed.

### Step 3 — finish
After the last file, emit exactly one `<done>` line:

```
<done summary="<one-line user-facing summary in {{userLocale}}>"
      filesCreated="N" filesUpdated="M" filesSkipped="K" />
```

## Rules

1. **Complete files only** — no placeholders, no TODOs, no `// rest unchanged`.
2. **Consistent naming** across all files; imports must resolve.
3. **Minimal but complete** — only essential files; no boilerplate that the
   user did not ask for.
4. **Best practices** — follow language / framework conventions visible from
   project meta (if provided) or industry defaults.
5. **Idempotent** — if invoked twice with identical instruction and existing
   files match, emit `op="skip"` for unchanged files rather than re-writing.
6. **Stream-friendly** — emit `<tree>` FIRST so the UI can show the plan
   before file contents start arriving. Each subsequent `<file>` must
   correspond to an entry in the tree (no out-of-tree files).
7. **No `<file>` content may itself contain the literal string `</file>`** —
   if your generated code needs it, escape as `&lt;/file&gt;`.

## File Selection Guidelines

- Include a package manager config (`package.json`, `pom.xml`, `go.mod`, etc.)
  when applicable.
- Include a `README.md` with setup instructions (under 80 lines).
- Include a main entry point.
- Include config files only when needed (`tsconfig.json`, `.env.example`,
  etc.).
- Split large modules into separate files.
- Include tests ONLY if explicitly requested.

## Output Language

- All comments, README copy, and `<done>` summary in {{userLocale}}.
- Code identifiers, file names, and CLI commands stay in English.
