# CodePilot IDEA 插件 — 总体架构设计

> 目标：在 JetBrains IDEA 中提供对标 Cursor 的 AI 编码体验（Chat / Agent / 重构 / 评审 / 注释 / 测试 / 文档），所有模型调用都经由自研 Java 后端中转，支持 MCP / Skill 商城、断点续跑、本地化会话存储与多语言场景。

---

## 2. 核心架构变化（对比初版设计）

| 维度 | 初版设计 | 当前实现 |
|---|---|---|
| **执行引擎** | AgentLoop（自实现while循环） | StateGraph编排（Spring AI Alibaba Graph，LangGraph Java port） |
| **Graph拓扑** | 单循环Plan→Act→Observe | 声明式DAG：intake→planning→preCheck→generate→gather→reenter→applyPatch→verify→repair→commit→finalize |
| **Plan展示** | 独立Plan面板（右侧面板） | 嵌入消息内（user_plan SSE事件→planSteps字段） |
| **Plan结构** | 单层Plan+TaskLedger | 双层：UserPlan（用户可见）+ ExecutionPhases（Graph内部驱动） |
| **SSE事件** | 扁平事件集 | 双层架构：底层(tool_call/tool_result_ack) + 语义层(agent_thinking/agent_reading/agent_writing/agent_running) |
| **工具回传** | HTTP POST同步等待 | ToolResultBus(Redis Pub/Sub + CompletableFuture双机制) |
| **前端UI** | Swing组件 | JCEF嵌入React/Vite WebUI（AgentStepCard/AgentContentRenderer等） |
| **Prompt模板** | 内联字符串 | PromptRegistry版本化文件（classpath:/prompts/*.txt） |
| **Deep Research** | 无 | 独立Graph拓扑：generate→gather→searchEvaluate→synthesize |
| **MCP/Skill集成** | 对话中路由 | Graph 已支持 `mcp.call`（`McpToolExecutor` + `userMcps`）；MCP Hub CRUD 与对话内工具治理 UI 仍分离 |

## 4. SSE事件双层架构

### 4.1 底层事件（协议层，机器消费）

| 事件 | 方向 | 用途 |
|---|---|---|
| `tool_call` | 后端→插件 | 请求客户端执行工具，含id/name/args |
| `tool_result_ack` | 插件→后端 | 工具执行结果回传（经由ToolResultBus） |
| `graph_plan` | 后端→插件 | Graph内部执行阶段（phases[]） |
| `graph_transition` | 后端→插件 | 节点跳转通知（from→to） |
| `graph_info_request` | 后端→插件 | 信息收集请求（GatherAction触发） |
| `graph_info_result` | 插件→后端 | 信息收集结果回传 |
| `user_plan` | 后端→插件 | 用户可见的Plan步骤 |
| `user_plan_progress` | 后端→插件 | Plan步骤状态更新 |

### 4.2 语义层（交互层，用户消费）

| 事件 | 方向 | 渲染方式 |
|---|---|---|
| `agent_thinking` | 后端→前端 | AgentStepCard(thinking)：💡 "我已分析你的需求" |
| `agent_reading` | 后端→前端 | AgentStepCard(reading)：📂 "CodePilot 读取了文件" |
| `agent_writing` | 后端→前端 | AgentStepCard(writing)：📝 "CodePilot 想要修改文件" |
| `agent_running` | 后端→前端 | AgentStepCard(running)：⌨️ "终端已运行" |
| `delta` | 后端→前端 | AgentContentRenderer：Markdown/plan/file/run结构化渲染 |
| `done` | 后端→前端 | 流结束信号，含reason和continuationToken |

### 4.3 前端渲染Pipeline

```
SSE事件 → onPluginEvent(bridge.ts) → App.tsx dispatch → ChatMessage更新
  ├── agent_thinking → agentSteps.push({type:'thinking',...})
  ├── agent_reading  → agentSteps.push({type:'reading',...})
  ├── agent_writing  → agentSteps.push({type:'writing',...})
  ├── agent_running  → agentSteps.push({type:'running',...})
  ├── user_plan      → planSteps=[{id,title,status}...] 嵌入消息内
  ├── delta          → content追加 → AgentContentRenderer解析XML标签
  │   ├── <plan>     → 高亮思考块
  │   ├── <file>     → 可折叠代码块
  │   └── <run>      → 可折叠命令块
  └── tool_call      → toolCalls=[{id,name,args,status}...]
```

## 5. 关键设计原则

1. **Graph Engine替代AgentLoop**：使用Spring AI Alibaba的StateGraph声明式编排，替代自实现的while循环。图在启动时编译一次，每次请求invoke。支持条件边（ConditionalEdges）实现动态路由。
2. **统一对话端点**：Chat与Agent共用`/v1/conversation/run`，由`mode`字段路由。Graph引擎在`policy.engine="graph"`时激活。
3. **双层Plan架构**：UserPlan（用户可见步骤，嵌入消息内）+ ExecutionPhases（Graph内部驱动循环）。PlanningAction的skipPlan机制允许LLM判断简单任务跳过详细规划。
4. **SSE事件双层分离**：底层（tool_call/tool_result_ack）供机器消费驱动工具执行；语义层（agent_thinking/reading/writing/running）供用户消费展示Agent行为。
5. **ToolResultBus异步回传**：CompletableFuture（单实例）+ Redis Pub/Sub（跨副本）双机制。等待线程注册Future后发tool_call，HTTP handler完成后publish。
6. **ContextStore占位符机制**：全量代码存CefChatPanel.contextStore Map，消息仅传`\u0001ctxId`引用，避免消息膨胀。
7. **ToolDispatcher线程安全**：读操作(fs.read/fs.list/fs.search)走ReadAction，写操作(fs.write/fs.replace/patch)走WriteCommandAction，确保IDE线程模型合规。
8. **后端不落库用户会话/代码**：仅做转发与编排，checkpoint仅Redis短TTL缓存。
9. **Prompt模板版本化**：18个必需段存在classpath:/prompts/下，PromptRegistry启动时加载，永不暴露给客户端。
10. **Deep Research模式**：独立Graph拓扑（generate→gather→searchEvaluate→synthesize），通过`policy.graphTemplate="deep-research"`激活。
11. **可中断+可恢复**：AskUser节点抛出GraphInterruptException中断图执行，保存checkpoint到Redis；resume时从checkpoint恢复状态继续执行。
12. **写操作必须经过Diff预览+用户确认**：ApplyPatchAction批量发tool_call，PatchApplier支持per-hunk Accept/Reject。
13. **MCP Hub已实现但未集成**：商城CRUD已完成，但对话中的MCP工具路由尚未接入Graph引擎。
14. **代码索引纯本地**：BM25(40%)+语义余弦(30%)+符号匹配(20%)+路径匹配(10%)混合评分。
15. **多文件Diff原子化**：Agent产出多文件Patch时，以MultiFileDiffPanel展示，支持per-hunk Accept/Reject。

## 6. Graph引擎拓扑

### 6.1 标准模式

```
START → intake ──→ planning ──→ preCheck ──→ generate ──→ applyPatch ──→ verify
              │         │           │           │            │           │
              │         ↓           ↓           ↓            ↓           ↓
              │      finalize    gather      gather       repair      repair
              │                  │           │            │           │
              │                  ↓           ↓            ↓           ↓
              │               reenter ──→ reenter     applyPatch   askUser
              │               │    ↑
              │               └────┘ (回originating节点)
              │
              └→ (resume时跳到目标节点)

commit ──→ preCheck (下一phase) / finalize (全部完成)
askUser ──→ 任意目标节点 / END
finalize ──→ END
```

### 6.2 Deep Research模式

```
START → intake → planning → generate → gather → reenter → searchEvaluate
                                                      │
                                              sufficient?
                                             /           \
                                           yes            no
                                           /                \
                                       commit          generate (重试)
                                       │
                               more phases?
                              /           \
                            yes            no
                           /                \
                      generate          synthesize → finalize → END
```

## 7. 跨平台命令执行

- 插件构建期注入目标OS标识（也可运行时检测`SystemInfo`）。
- 后端在生成`shell`工具调用时附加`osHint`，由插件端做最终命令适配。
- 强制超时（默认60s）、输出截断（默认64KB）、危险命令黑名单。
- 失败修复：把`stderr + exitCode`回传模型，最多重试N=3。

## 8. 上下文与提问交互

### 8.1 MNI与分层上下文

每次请求只带"本轮所需最小集"：`pinned + ledgerCursorBranch + lastPlanDigest + lastAssistantTurnSummary(≤400 tokens) + sessionDigest? + recent(M条) + refs(path+range+sha1) + completedToolCallsTail(K=5) + earlierToolCallsCount`。

客户端先做预算与折叠；服务端`ContextBudgeter`做二次裁剪并按层级映射到system/user/tool消息。预算仍不足时，追加compact段让模型本轮先输出digest再继续。

### 8.2 AskUser交互（替代原needs_input）

AskUserAction是Graph的一个节点，当模型需要用户确认时：
- 抛出GraphInterruptException中断图执行
- 发送needs_input SSE事件到前端
- 保存checkpoint到Redis（含continuationToken和nextNode）
- 用户回答后，resume从checkpoint恢复，路由到目标节点继续执行

## 9. 断点恢复

- GraphInterruptException触发checkpoint保存（Redis TTL 24h）。
- 插件端收到done事件后，保存continuationToken。
- Resume时：加载checkpoint → 合并用户答案 → 从nextNode继续执行图。
- 副作用幂等：每个工具调用持久化toolCallId，恢复时跳过已执行步骤。

## 10. 升级与状态重置

- **后端system Skill/提示词**：通过配置中心推送，PubSub失效缓存即生效。
- **插件本体**：双轨升级——热更新（不重启IDE，替换JCEF前端资源/模板缓存）+ 重启更新（涉及Kotlin类装载变更时）。
- **强制状态重置**：`CodePilot.Reset.*`命令族 + sentinel文件机制，即使IDE卡死也可通过文件通道触发干净重启。
