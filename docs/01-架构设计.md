# CodePilot IDEA 插件 — 总体架构设计

> 目标：在 JetBrains IDEA 中提供对标 Cursor 的 AI 编码体验（Chat / Agent / 重构 / 评审 / 注释 / 测试 / 文档），所有模型调用都经由自研 Java 后端中转，支持 MCP / Skill 商城、断点续跑、本地化会话存储与多语言场景。

---

│             CodePilot Backend  (Spring Boot 3 + Spring AI)         │
│                                                                  │

## 2. 模块划分

### 2.1 插件端（IntelliJ Platform Plugin）
| 模块 | 职责 |
|---|---|
| `ui-chat` | ToolWindow 聊天面板，支持流式渲染、Diff 预览、可中断 |
| `ui-action` | 右键菜单、快捷键 Action（重构/评审/注释/测试/文档） |
| `ui-settings` | 模型配置、商城、会话目录、快捷键、Telemetry 开关 |
| `context` | 选中代码 / 文件 / 多文件 / 工程上下文采集（含 PSI 摘要） |
| `tool-fs` | 读/写/替换/删除/创建/移动 文件，统一走 VFS + Diff 预览 |
| `tool-shell` | 跨平台命令执行（win/mac/linux 自适配），结果回灌 |
| `session` | 本地隐藏目录的会话存储、列表、续聊、断点恢复 |
| `mcp-client` | 与后端 MCP Hub 交互，本地拉起 stdio MCP（可选） |
| `transport` | HTTP/SSE 客户端，鉴权、重试、指纹绑定 |

会话本地目录（默认）：
- macOS / Linux: `~/.codePilot/sessions/<workspaceHash>/<sessionId>/`
- Windows: `%USERPROFILE%\.codePilot\sessions\<workspaceHash>\<sessionId>\`
- 可在 Settings → CodePilot → Session Path 修改。

每个会话目录结构：
```
<sessionId>/
├── meta.json          # 标题、模型、模式、创建/更新时间
├── messages.ndjson    # 每行一条消息，便于追加/恢复
├── steps.ndjson       # Agent 模式下的步骤、工具调用、结果
├── plans/             # 规划版本：plan-{n}.json（含步骤状态机）
├── ledger.json        # 任务清单（goal/subtasks/cursor/notes/blockers）
├── summaries/         # 历史压缩摘要：digest-{ts}.json
├── checkpoints/       # 断点：未完成的 plan / 已应用 diff 列表 / completedToolCalls
└── attachments/       # 用户拖入的临时文件副本（可选）
```

### 2.2 后端（Java 17 + Spring Boot 3 + Spring AI）
| 模块 | 职责 |
|---|---|
| `gateway` | 统一鉴权、限流、签名校验、SSE 透传 |
| `chat-service` | 问答模式：Prompt 编排 + 流式输出，仅基于上下文，不写文件 |
| `agent-service` | Agent 模式：单次调用让模型先 Plan 再 Act 再 Observe，工具路由 |
| `prompt-orchestrator` | 在一次模型调用里串联 Plan→（按需）Compact→Execute 多段式 system，不暴露独立接口 |
| `action-service` | 重构 / 评审 / 注释 / 测试 / 文档：定义化 Prompt + 输出契约 |
| `tool-router` | 把 Agent 的 ToolCall 路由到 FileOps / Shell / MCP / RAG / Skill |
| `model-service` | 模型注册表：内置模型 + 自定义服务商（OpenAI 协议兼容） |
| `mcp-hub` | MCP/Skill 商城后端：清单、版本、签名、安装包分发（仅元数据） |
| `rag-service` | 临时项目级向量索引（按 sessionId TTL，使用 Spring AI VectorStore） |
| `prompt-registry` | Prompt 模板版本化、灰度、A/B |
| `audit` | 全链路审计（脱敏后落库），不存原始代码内容 |

## 3. 关键设计原则

1. **统一对话端点**：Chat 与 Agent 共用一个输入框、一个 SSE 端点 `/v1/conversation/run`，仅以 `mode="chat"|"agent"` 路由不同的 PromptOrchestrator 策略；SSE 事件集合一致，`plan/plan_delta/tool_call/self_check/needs_input/risk_notice/task_ledger` 在 chat 模式下不出现。
2. **多轮多次往返**：一次 `/conversation/run` 只推进任务的一小段（如"先读规范"），命中自然停顿（subtask_done / awaiting_user_input / paused / max_steps）即结束；插件保留 `plan + taskLedger + completedToolCalls + sessionDigest` 到本地，以 `intent="continue"|"answer"` + `continuationToken` 发起下一次 run，模型据此续推。后端不持久化任何会话内容。
3. **Task Ledger（任务清单）作为一等公民**：模型与用户共同维护 `goal / subtasks / cursor / notes / blockers`；面板与 Plan 面板并列展示；可勾选/编辑/暂停。
4. **MNI（最小必要信息）+ 分层上下文**：每次请求只携带"完成本轮任务必需"的最小集合：`MUST-KEEP（pinned/ledger.cursorBranch/decisions）+ lastAssistantTurnSummary + lastPlanDigest + sessionDigest? + recent(M条) + refs（仅 path+range+sha1）+ completedToolCallsTail(K=5) + earlierToolCallsCount`。客户端先做 MNI 装配与本地预算，服务端 ContextBudgeter 再做二次裁剪；预算仍紧张时 Prompt 内追加 compact 段，让模型先输出 digest 再继续。
5. **结构化追问 + 多种回答方式**：模型识别信息不足时通过 `needs_input` 给出 1–3 个编号问题（带选项与推荐解法）；用户可点选、输入"编号+答案"、或完全自由输入。回答打包到 `answers[]`，下次以 `intent="answer"` 提交，禁止重复追问已回答内容。
6. **后端不落库用户会话/代码**：仅做转发与编排，必要时仅在内存/Redis 短 TTL 缓存。
7. **统一工具协议**：插件端 FileOps / Shell 与后端 MCP Tool 使用同一 JSON Schema，便于 Agent 统一调度。
8. **可中断 + 可恢复**：每一步 Agent 操作生成 `step` 记录与 `checkpoint`；恢复时携带 `lastPlan + completedToolCalls + taskLedger + sessionDigest`，从最后一个未完成 cursor 重放。
9. **写操作必须经过 Diff 预览 + 用户确认**（可配置自动确认白名单），高风险工具调用必须先发 `risk_notice`。
10. **工具调用后必须自检**：`policy.selfCheck=true` 时模型在拿到工具结果的下一轮回复中先输出 `selfCheck`，决定 `continue|retry|replan|finalize|ask_user`。
11. **歧义优先澄清**：`policy.askPolicy="prefer-ask"` 时模型在不确定方案时发 `needs_input` 让用户选，不擅自决策。
12. **Prompt 与代码解耦**：所有 Prompt 集中在 `prompt-registry`，支持热更新与按语言/任务分发。
13. **骨架 workflow + Skill 混合 Prompt 架构**：基础契约（envelope/工具/上下文预算/交互/安全）作为"骨架 workflow"每轮加载；领域知识（语言、风格、修复、场景、Action）以 **Skill 包**形态按需激活，分 **system / user** 两域，运行时严格区分。
14. **Skill / MCP 双域分治**：system 仅驻后端、**永不下发**（API 仅返回安全摘要）；商城分发链路只服务 user 域。user Skill / MCP 二选一作用域：**project（落项目根 `<projectRoot>/.codePilot/`）** 或 **global（落系统配置目录 `~/Library/Application Support/CodePilot`、`~/.config/codePilot`、`%APPDATA%\CodePilot`）**；project 优先于 global。运行时通过 `[USER_SKILL_BEGIN]…[USER_SKILL_END]` 隔离，不允许 override system，不允许重定义 envelope/guard。
15. **系统提示词机密性**：所有 system 提示词、Skill 段、orchestrator 装配信息均不可外泄，前置（输入意图检测）+ 后置（响应流泄露过滤 SystemPromptLeakFilter）双重防护，命中即整轮替换为标准拒绝并审计。
16. **插件自更新优先热更新**：提示词模板、UI/前端资源、词表/i18n 走热更新不重启 IDE；涉及类装载或 IDE SDK 的变更自动回退为重启更新；签名校验、通道、灰度、回滚机制齐备，system Skill 正文不在客户端任何形态持久化。
17. **强制状态重置可回收**：插件提供 `CodePilot.Reset.*` 系列命令与文件 sentinel 通道，即使 IDE 卡死仍可触发"干净重启"——启动时跳过损坏的本地状态，仅以内存默认值启动，旧数据重命名保留到 `*.broken-<ts>/` 以便取证。
18. **OpenAI 协议为最小公分母**：自定义服务商只需提供 `baseUrl + apiKey + model`。

## 4. 模式说明

### 4.1 Chat 模式（问答）
- 只读上下文：选中代码 / 文件 / RAG 检索结果。
- **后端禁用**所有写文件 / 执行命令工具。
- 适合解释、答疑、代码理解。
- 同样接入 ContextCompactor：当历史 tokens 超阈值时自动总结成 digest。

### 4.2 Agent 模式（Plan-First + 多轮增量推进）
- **规划是 Prompt 行为**：每次 `/conversation/run`（mode=agent）调用，模型在同一次回复里先输出 `plan/planDelta`、`taskLedger`，再按 cursor 输出至多一个 `toolCall` 或 `final`。
- **一次 run 只推进一段**，命中以下任意条件即返回 `done`，等下一次 run 续推：完成一个子任务（`final.subtaskDone=true`）/ 发出 `needs_input` 等待用户回答 / 命中 `policy.maxSteps` / 被 `/conversation/stop` / 同步骤连失败 3 次。
- 主循环（同一 SSE 通道内多次模型调用）：
  `(Plan|PlanDelta + Ledger?) → ToolCall → Tool 执行 → ToolResult → SelfCheck → continue|retry|replan|finalize|ask_user → 下一次模型调用 ...`
- Plan / Ledger 由模型增量更新；插件端按版本号合并展示。
- 用户可在 Plan / Ledger 面板里勾选/编辑/暂停/追问；这些编辑通过 `userPlanEdits` / `intent="answer"` 在下次请求里送回。
- 写操作产出 `Patch`，由插件端 IDE 内 Diff 预览；高风险操作先发 `risk_notice` 卡片。
- 命令失败：`selfCheck` 先分类（TOOL/CODE/ENV/SPEC），再决定最小修复或 replan；连失 3 次进入 `needs_input`。

### 4.3 规划能力（纯 Prompt 实现）
- 通过 `prompt.agent.system` 强约束：每次输出必须包含 `plan` 字段（首轮全量、后续可只发 `planDelta`）。
- 输出严格 JSON Schema 校验（后端 `BeanOutputConverter`），不合规则触发一次自我修正重发。
- 规划与执行可使用**同一模型**；如需更强规划能力，可在 `policy.plannerHint` 里指定 `reasoningHigh`，由后端把同一模型的推理力度调高（如 `reasoning_effort=high`）。

### 4.4 对话总结能力（在同一次 Prompt 中完成）
- 不再单独提供 `/session/compact` 接口。后端 `PromptOrchestrator` 在拼装 Prompt 时检测：
  - 估算 tokens > 模型上下文 70%；
  - 消息条数 > 80 或 Agent 步骤 > 12；
  - 用户在面板手动点击"压缩对话"。
- 命中后，本次模型调用的 system 会**追加一段压缩指令**，要求模型先输出 `digest`（写入回复 JSON 的 `digest` 字段），再输出 `plan/planDelta` 与 `toolCall`，**单次调用一次完成**。
- 插件端收到带 `digest` 的回复后，把早期消息折叠为 `system: <digest>`，仅保留最近 N 条与 digest 之后的新消息。

## 5. 跨平台命令执行

- 插件构建期注入目标 OS 标识（也可运行时检测 `SystemInfo`）。
- 后端在生成 `shell` 工具调用时附加 `osHint`，由插件端做最终命令适配：
  - Windows → `cmd /c` 或 `powershell -Command`
  - macOS / Linux → `/bin/bash -lc`
- 强制超时（默认 60s）、输出截断（默认 64KB）、危险命令黑名单（如 `rm -rf /`、`format`、`shutdown`）。
- 失败修复：把 `stderr + exitCode` 作为 Observation 回传模型，最多重试 N=3。

## 6. 上下文与提问交互（贯穿 Chat / Agent）

### 6.1 MNI 与分层上下文
- 每次请求只带"本轮所需最小集"：`pinned + ledger.cursorBranch + lastPlanDigest + lastAssistantTurnSummary (≤400 tokens) + sessionDigest? + recent(M条) + refs(path+range+sha1) + completedToolCallsTail(K=5) + earlierToolCallsCount`。
- 客户端先做预算与折叠（多条 recent → localDigest）；服务端 `ContextBudgeter` 做二次裁剪并按层级映射到 system/user/tool 消息。
- 预算仍不足时，追加 `prompt.agent.compact` 段，让模型在**同一次回复**里先输出 `digest` 再继续，不新增接口。
- 客户端持久化模型回传的 `summaryForNextTurn` 与 `hintsForContext.pin/unpin/requestRead`，形成"双向记忆"。

### 6.2 结构化追问（needs_input）
- 模型识别信息不足/高风险授权/歧义时发 `needs_input`：
  - 每题带编号 `index`、可选 `options[]`（含 `impact / pros / cons`）、`defaultOptionId`（安全推荐）；
  - `kind` 支持 `single-choice | multi-choice | yes-no | freeform`；
  - `freeformAllowed=true` 允许用户完全自由输入；
- 用户回答三种方式，统一打包 `answers[]`：
  1) 点选选项 → `{questionId, optionId}`
  2) 编号+答案 → 插件解析 `1: b; 2: a; 3: 600` 等格式
  3) 完全自由输入 → `{freeform: "..."}`（插件尽力匹配 questionId，无匹配则整体提交，下一轮模型据此重规划）
- 重复问同一问题的请求会被服务端拦截并触发一次自动纠错；已回答内容写入 `taskLedger.notes`，禁止再被追问。

### 6.3 多轮往返的工作模式
- 一次 run 常常只完成一件子任务（如"读规范"）即 `done.reason=subtask_done`；
- 用户点击"继续"或直接追加输入 → `intent=continue`；
- 用户回答 `needs_input` → `intent=answer`；
- 需要从中断处恢复 → `/conversation/resume`（等价 `/conversation/run` + 续跑上下文）。

### 6.4 Skill 体系（骨架 workflow + 按需 Skill 加载，system / user 双域）
- **骨架 workflow**（每轮必加载）：身份与红线、Plan-First envelope、工具 Schema、selfCheck、上下文预算、needs_input、riskNotice、注入防御。
- **system Skill / MCP**：**严格驻后端，永不下发到客户端**。后端 `SkillRouter` 在拼装 system 段时读取后端内置或配置中心下发的版本；商城仅对 system 返回"安全摘要"（id / version / title / triggersBrief / permissionsBrief / audit / changelogUrl），正文永不可通过任何 API 取得；因此 system 正文可随后端发版/配置中心随时热更新，客户端无须感知。
- **user Skill / MCP**：完全位于**用户侧**，后端不持久化。用户可以：
  - 从商城安装他人分享的 Skill / MCP（下载后在用户本地落地，作用域由用户选择）；
  - 也可自己在插件内新建 / 导入。
- **user 作用域：project vs global（二选一）**：
  - **project（项目生效）**：安装路径在**项目根**的隐藏目录 `<projectRoot>/.codePilot/skills/` 或 `<projectRoot>/.codePilot/mcps/`；仅当用户在该工程中工作时生效，可随工程一起 git 共享（是否纳入版本控制由团队决定）。
  - **global（全局生效）**：安装路径在**系统配置目录**，对当前登录用户的所有工程生效：
    - macOS：`~/Library/Application Support/CodePilot/skills/`、`~/Library/Application Support/CodePilot/mcps/`
    - Linux：`~/.config/codePilot/skills/`、`~/.config/codePilot/mcps/`（遵循 XDG Base Dir）
    - Windows：`%APPDATA%\CodePilot\skills\`、`%APPDATA%\CodePilot\mcps\`
  - 冲突时 `project > global`；重复 id 时 project 作用域覆盖 global（仅做 `merge: append|wrap`，不允许 `override` system）。
- **运行时严格区分**：每次 `/conversation/run` 由插件把"本轮命中的 user Skill"打包进 `userSkills[]` 上行并带 `scope: project|global`；后端仅本次请求使用，用完即丢弃；所有 user Skill 段在 system 拼装时被包裹为 `[USER_SKILL_BEGIN id=... source=user scope=project|global]…[USER_SKILL_END]`；user Skill 不允许 `merge: override` 任何 `source=system` 的 Skill；不允许重定义 envelope/guard 协议；权限受工具白名单约束。
- **市场与安全**：MCP/Skill 商城的分发体系只对 **user Skill / MCP** 起作用——下载产物落到用户本地所选作用域（project 或 global）；**商城不分发 system 正文**，system Skill 仅在后端出现。第三方包仍需平台签名，安装后自动标记 `source=user`。
- **系统提示词机密性**：模型层有"绝不外泄"硬约束（Prompt §0/§10）；后端在响应流必经 `SystemPromptLeakFilter`，命中即整轮替换为标准拒绝并审计 `45002`。

### 6.5 Skill / MCP 安装与来源渠道（四种来源 × 两种作用域）

> **范围说明**：本节规则只适用于 **user 域**（`source=user`）。**system 域永远只在后端，客户端没有任何入口可以"安装 system Skill / MCP"**；商城里看到的 system 条目是只读摘要。

**四种来源**：

1) **官方商城**（`registry=official`）：CodePilot 平台官方注册中心（对应后端商城）；所有包必须经平台签名；安装前自动通过 `signature` 与 `sha256` 双重校验。
2) **第三方市场**（`registry=third-party`）：用户可在 Settings 中添加一个或多个外部注册中心（Registry URL，见接口 §7.9）；每个外部 registry 必须声明自己的公钥，插件端据此验证下载包签名；未通过签名验证的包**拒绝安装**，除非用户在高危确认对话框显式"信任此 registry"。
3) **本地文件安装**（`registry=local`）：用户直接从本地磁盘导入 `*.cps`（CodePilot Skill Pack）或 `*.mcp` 文件（本质是带 manifest 的 zip）；常用于团队离线分发或私有分享。
4) **直接新建**（`registry=builtin-ide`）：由插件内的"新建 Skill / 新建 MCP"向导直接在作用域目录下创建 skill.yaml / mcp.yaml，不经过任何 registry，默认**未签名**（仅本人可用；导出给他人需先签名）。

**两种作用域**（与 §6.4 一致）：
- `project`：装到 `<projectRoot>/.codePilot/{skills,mcps}/installed/<id>@<version>/`
- `global`：装到用户配置目录（macOS/Linux/Windows 各自约定路径）

**统一的安装步骤（客户端 PackageInstaller）**：

```
1. resolve       通过 registry 的 manifest 接口获取 { id, version, source=user, sha256, signature, permissions, triggersBrief, downloadUrl, dependencies[], license }
2. preflight     检查：minPluginVersion、OS 兼容、工具权限是否超出当前白名单、依赖的其他 Skill/MCP 是否齐全
3. verify        下载并校验 sha256；按 registry 的公钥校验 signature；未通过 → 终止并提示
4. consent       弹出高危确认框：显示权限声明（fs/shell/网络/MCP 外呼）、triggers 摘要、签名主体、license；用户明确勾选"理解并允许"
5. scope-pick    用户在 project / global 二选一（由安装按钮上下文决定默认值）
6. extract       解压到所选作用域的 `installed/<id>@<version>/` 目录，同时更新 `index.json`
7. activate      重载 SkillRegistry（客户端本地索引）；MCP 如需子进程，按 manifest 启动子进程
8. report        向后端 /audit/event 上报 { action=install, source=user, registry, id, version, scope, sha256 }（不含 yaml 正文）
```

**来源可信度分级**：
- `official` → trust=high（默认允许自动安装依赖）
- `third-party` 且已"信任此 registry" → trust=medium（依赖安装仍需用户确认）
- `local` 与 `builtin-ide` → trust=low（仅本人可见；导出需签名）
- 任意来源的未签名包一律 trust=untrusted，默认禁用，需用户明确放行。

**冲突与优先级**（配合 §6.4）：
- 来源不影响优先级，仅影响可信度；执行时仍按 `source(system>user) > priority > scope(project>global>system) > version` 排序。
- 同 id 多来源安装时，在商城 UI 里以"主版本卡片 + 其他来源候选"的形式并列展示；用户显式选择启用哪一条。

**卸载、更新、导出**：
- 卸载：删除作用域目录下的 `installed/<id>@<version>/` 并从 `index.json` 移除；可选"软卸载"仅加入 `disabled.json`。
- 更新：重新走 1–8 步；更新后仅保留最新版本，旧版本移到 `~/.codePilot/trash/<id>@<oldVersion>/` 保留 14 天；
- 导出：将某个 user Skill 打包为 `*.cps` 并生成用户级签名（由用户的本地密钥签），方便分享；**任何情况下都不会把 system Skill 导出**。

## 7. 断点恢复

- 每个 Agent 步骤入栈到 `steps.ndjson`，并在关键节点写 `checkpoints/<n>.json`。
- 用户中断或异常退出后：
  1. 重新打开会话；
  2. 插件端读取最后一个 checkpoint；
  3. 把 `已完成步骤摘要 + 未完成 plan` 重新提交后端 `/conversation/resume`；
  4. 后端基于摘要继续推理，不重复已成功的副作用步骤。

## 8. 升级与状态重置（端到端）

- **后端 system Skill / 提示词**：通过配置中心或 marketplace.system 推送，PubSub 失效缓存即生效；客户端无须感知，因为它从不持有正文。
- **插件本体**：双轨升级——
  - **热更新**（不重启 IDE）：仅替换提示词模板缓存、ToolWindow 前端资源、tokenizer 词表、i18n 等"无类装载副作用"的资源；
  - **重启更新**：涉及 Kotlin 类、Action/Intention 注册、native lib 时回退此路径；写到 `pluginsTemp/` 由 IntelliJ `PluginInstaller` 在重启时替换；
  - 通道（stable/beta/dev/corp.\*）+ 灰度 + 签名校验 + 一键回滚（详见插件端 §14）。
- **强制状态重置**：插件内置 `CodePilot.Reset.*` 命令族；同时通过 `~/.codePilot/.flags/*` sentinel 文件提供"IDE 卡死时也可触发"的恢复路径——下次启动前由 `StartupActivity` 检测后跳过损坏的本地状态、以内存默认值启动、把可疑目录重命名为 `*.broken-<ts>/`（详见插件端 §15）。
- **数据安全**：升级、重置都不会上传/泄露任何 system Skill 正文；不会动用户会话目录、user Skill / MCP，除非用户在确认对话框中显式勾选清理。
