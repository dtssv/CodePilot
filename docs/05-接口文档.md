# 05 — 后端接口文档（v1）

## 0. 通用约定

- BaseURL：`https://api.codePilot.<corp>/v1`
- 编码：`UTF-8`，`Content-Type: application/json`
- 认证：`Authorization: Bearer <jwt>`
- 设备 & 签名头：
  - `X-CodePilot-Device-Id`：设备指纹（插件首启生成，本地保存）
  - `X-CodePilot-Ts`：unix 毫秒
  - `X-CodePilot-Nonce`：UUID
  - `X-CodePilot-Signature`：`HMAC_SHA256(deviceSecret, body + ts + nonce)`
- 流式：使用 SSE（`Accept: text/event-stream`），事件名见各接口。
- 统一响应（非流式）：
  ```json
  { "code": 0, "message": "ok", "data": { ... }, "traceId": "..." }
  ```
- 错误码（节选）：
  - `0` 成功 / `40001` 参数错误 / `40101` 未认证 / `40301` 越权
  - `42901` 限流 / `45001` 安全拦截 / `45002` 系统提示词外泄拦截 / `45003` user Skill 校验失败 / `50001` 内部错误 / `50002` 模型上游错误

## 3. 对话接口（Chat / Agent 共用）

> 插件端只有**一个输入框**与**一个模式开关**。`mode="chat"` 与 `mode="agent"` 共用同一组接口与 SSE 事件结构，仅在"是否允许工具调用与写操作"上有差异。下游通过 `mode` 路由到不同的 PromptOrchestrator 装配策略。

；
- 命中多段：支持分号/换行分隔；
- 命中失败：打包为 `freeform` 原文 + `confidence=0`；
- 用户输入里若包含"重新规划 / replan / 重来"等关键词，同时在下一次请求的 `policy.replanHint=true`。

模式差异（同一 schema，不同行为）：

| 维度 | `mode=chat` | `mode=agent` |
|---|---|---|
| 是否注入工具 schema | 否（`tools` 强制为空） | 是，按 `tools` 白名单 |
| 是否允许 `tool_call` 事件 | 否 | 是 |
| 是否允许 `plan / plan_delta` | 否 | 是 |
| 写文件 / 执行命令 | 禁止 | 允许（受插件端护栏） |
| `digest` 事件 | 允许（仅压缩历史） | 允许 |
| 多轮模型调用 | 单轮 | 多轮直到 `final` 或 `max_steps` |

### 3.2 SSE 事件（chat / agent 共用一组事件名）

| event | 何时出现 | data |
|---|---|---|
| `digest` | 命中上下文压缩条件 | 见后端 §6.2 `Digest` 结构 |
| `skills_activated` | 本轮 SkillRouter 激活的 Skill 集合（用于审计与调试） | `{"items":[{"id":"skill.lang.java","version":"1.2.0","tokens":260,"priority":50}],"droppedDueToBudget":[]}` |
| `task_ledger` | 模型更新任务清单（新增/完成/调整子任务、推进 cursor、补充 notes） | `{"goal":"...","subtasks":[...],"cursor":"t2","notes":[...]}` |
| `plan` | 仅 agent，首轮或 replan 时 | 完整 Plan JSON |
| `plan_delta` | 仅 agent，后续轮 | `{ "ops":[{"op":"add|modify|skip|markStatus", ...}] }` |
| `tool_call` | 仅 agent，模型选择的工具调用，需插件本地执行 | `{"id":"tc-1","name":"fs.read","args":{...},"riskLevel":"low","why":"读规范以决定后续步骤"}` |
| `tool_result_ack` | 仅 agent，后端已收到客户端回传的工具结果 | `{"toolCallId":"tc-1","status":"received"}` |
| `self_check` | 工具调用后的模型自检结论（见 Prompt §9.6） | `{"toolCallId":"tc-1","ok":true,"checks":[...],"nextAction":"continue|retry|replan|finalize|ask_user"}` |
| `needs_input` | 模型主动请求用户澄清/选择/授权 | 见 §3.2.1，结构化追问 |
| `risk_notice` | 即将执行高风险操作前的预警卡片（写/删/移/shell） | `{"toolCallId":"tc-9","kind":"shell|fs.delete|fs.move","preview":"...","mitigations":[...]}` |
| `delta` | 流式文本（chat 全程；agent 在 final 阶段或解释阶段） | `{"text":"..."}` |
| `patch` | 模型直接产出的 Patch（多见于 Action 类一键调用） | Patch JSON |
| `usage` | token 用量统计 | `{"promptTokens":1234,"completionTokens":234}` |
| `error` | 错误 | `{"code":...,"message":"..."}`<br>常见 code：`45001` 安全拦截、`45002` 系统提示词外泄拦截、`45003` user Skill 校验失败、`42901` 限流 |
| `done` | 整次结束（本次 run 推进到自然停顿） | `{"reason":"final|subtask_done|awaiting_user_input|paused|max_steps|stopped|failed","continuationToken":"ctk-..."}` |

> Agent 模式每"轮"事件顺序固定：`(digest?) → (task_ledger?) → (plan|plan_delta) → (risk_notice?) → (tool_call|delta) → (tool_result_ack) → (self_check?)`，多轮重复直到本次 run 命中 `policy.stopOn` 或 `final`。
> Chat 模式只会出现：`(digest?) → delta* → usage → done`。
> 收到 `done.reason="awaiting_user_input"` 后，插件应在输入框置入"针对该问题的回答"占位，并在用户回答时以 `intent="answer"` + 上一次的 `continuationToken` 调用 `/conversation/run`。

#### 3.2.1 `needs_input` 事件 Schema（结构化追问 + 建议解法）

```jsonc
{
  "title":"需要你确认几件事，我再继续",
  "reason":"当前目标下有 2 处方案分歧，误选会导致返工",
  "blocking": true,                              // true → done.reason=awaiting_user_input
  "maxAnswers": 3,                                // 期望的最大回答数
  "freeformAllowed": true,                        // 允许完全自由输入（不走选项）
  "questions":[
    {
      "id":"q1",
      "index":1,                                 // UI 展示编号；插件端按此解析 "1: b"
      "prompt":"是否允许修改 OrderService.createOrder 的公共签名？",
      "why":"实现幂等键最简方案需要新增一个参数",
      "kind":"single-choice",                    // single-choice | multi-choice | yes-no | freeform
      "required": true,
      "defaultOptionId":"b",                     // 推荐答案（UI 里高亮）
      "options":[
        {"id":"a","label":"允许（新增默认值参数）","impact":"low","pros":["改动最少"],"cons":["破坏接口稳定性"]},
        {"id":"b","label":"不修改签名，使用包装方法","impact":"low","pros":["API 稳定"],"cons":["代码略多"]},
        {"id":"c","label":"通过请求头 / MDC 透传幂等键","impact":"medium","pros":["无侵入业务代码"],"cons":["需要网关配合"]}
      ]
    },
    {
      "id":"q2",
      "index":2,
      "prompt":"使用什么存储去重键？",
      "kind":"single-choice",
      "required": true,
      "options":[
        {"id":"a","label":"Redis SETNX + TTL","impact":"low"},
        {"id":"b","label":"DB 唯一索引","impact":"medium"},
        {"id":"c","label":"二者结合（先 Redis 再 DB 兜底）","impact":"medium"}
      ]
    },
    {
      "id":"q3",
      "index":3,
      "prompt":"幂等窗口时长（秒）？",
      "kind":"freeform",
      "required": false,
      "placeholder":"如 600"
    }
  ],
  "notesForUser":[
    "点击任一卡片即可作为答案",
    "也可以在输入框输入：1: b; 2: a; 3: 600",
    "也可以直接写你的想法，我会据此重新规划"
  ]
}
```

插件端三种回答方式（都打包到下一次请求的 `answers[]`）：

- **点选**：`{"questionId":"q1","optionId":"b"}`
- **编号+答案**：用户输入 `1: b; 2: a; 3: 600`
  → 插件解析为 `[{"questionId":"q1","optionId":"b"},{"questionId":"q2","optionId":"a"},{"questionId":"q3","freeform":"600"}]`
- **自由输入**：`"用 Redis SETNX，TTL 10 分钟；签名保持不变"`
  → 插件尽力匹配到对应 questionId，匹配不到则作为 `{"freeform":"..."}` 整体提交，模型下一轮会据此重新规划。

### 3.3 `POST /conversation/tool-result`

  "sessionId":"sess-uuid",
  "toolCallId":"tc-1",
  "ok":true,

### 3.4 `POST /conversation/stop`
中断当前运行：`{ "sessionId":"sess-uuid" }`

### 3.5 `POST /conversation/resume` (SSE)
基于插件本地 plan/checkpoint 续跑（chat 模式不需要恢复，仅 agent 使用）：
```json
{
  "sessionId":"sess-uuid",
  "mode":"agent",
  "modelId":"codePilot-pro",
  "input":"继续之前的任务",
  "lastPlan": { /* 插件本地最近一次 plan/planDelta 合并后的完整 Plan */ },

    {"toolCallId":"tc-1","stepId":"s1","ok":true},

}
```

## 4. Agent 模式补充说明

本章仅作为对 §3 的补充，**不再暴露独立接口**。所有调用统一走 `/conversation/*`（mode=agent）。

- **规划是 Prompt 行为**：模型在每轮回复中先输出 `plan`（首轮或 replan）或 `plan_delta`（后续轮），不需要独立 `/plan/*` 接口。
- **上下文压缩是 Prompt 行为**：命中阈值时模型在同一次回复里先输出 `digest`，不需要独立 `/session/compact` 接口。
- **工具结果自检是 Prompt 行为**：在 `policy.selfCheck=true` 时，模型在拿到 `tool-result` 的下一轮回复中先输出 `self_check`（见 Prompt §9.6），再决定 `continue | retry | replan`。
- **断点续跑**：使用 `/conversation/resume`（mode=agent），把插件本地 `lastPlan + completedToolCalls + sessionDigest` 送回后端，模型在同一 SSE 通道内继续推进。
- **可中断**：用 `/conversation/stop` 中断；插件端会把"中断时的 plan + 最后成功的 toolCallId"落到本地 `checkpoints/`，供下次 resume 使用。
- **跨 OS 命令执行**：后端在 `tool_call` 中附带 `osHint`；由插件端做最终命令适配，结果回传结构见 §3.3。

### 7.7 Skill 类型补充说明
- 当 `type=skill` 时：
  - `manifest` 即 Skill 包 yaml（见 Prompt §A.2，必须含 `source: system|user`）；
  - **system Skill 永不下发**：商城对 `source=system` 的 Skill 仅暴露"安全摘要"（不含 `systemPrompt / examples / tests` 正文）。下方 §7.8 列出可返回的字段；
  - **user Skill 下载与安装均落用户本地**（`~/.codePilot/skills/installed/...`），后端不存放任何 user Skill 内容，仅记录"用户安装了 X@Y 版本"的元数据用于商城重装与统计；
  - `download` 内容仅对 `source=user` 提供，且**不允许包含运行时代码**；
  - 第三方 Skill 必须通过平台签名（`signature`）；安装后默认 `source=user`，仅在本地生效；
- 客户端在每次 `/conversation/run` 时把"本轮可能命中的 user Skill"打包进 `userSkills[]` 上行（见 §3.1.2）；后端**仅本次请求使用**，不落库、不缓存原文。

### 7.8 System Skill 只读 API（仅元数据，禁止下发正文）

`GET /mcp/packages/{slug}` 与 `GET /mcp/packages/{slug}/versions/{version}/manifest` 在 `source=system` 时**强制裁剪**为安全摘要：

```jsonc
{
  "id": "skill.lang.java",
  "version": "1.3.0",
  "source": "system",
  "title": "Java language profile",
  "scope": "builtin",
  "owner": "codePilot-core",
  "triggersBrief": [
    {"language": ["java"]},
    {"fileGlob": ["**/*.java","**/pom.xml","**/build.gradle*"]}
  ],
  "permissionsBrief": {
    "tools": ["fs.read","fs.replace","fs.outline","shell.exec"],
    "risk":  ["low","medium","high"]
  },
  "audit":   { "tokensEstimate": 260 },
  "changelogUrl": "https://docs.codePilot.<corp>/skills/skill.lang.java/CHANGELOG"
}
```

约束：
- **永远不会出现** `systemPrompt / examples / tests / merge / dependsOn` 正文级字段；
- `download` 端点对 `source=system` 直接 `403 forbidden` 并审计；
- `triggersBrief` 仅给出"够展示与排序"的子集，避免泄露完整触发表达式；
- `permissionsBrief` 用于商城展示；运行时白名单以服务端为准。

> 这样设计的目的：system Skill 的正文随后端发版/配置中心实时更新，客户端无须感知；用户既能看到"这个 Skill 在做什么、需要什么权限"，也无法通过任何接口拿到完整提示词。

### 7.9 Third-party Registry Protocol（第三方市场对接协议）

第三方市场只需暴露下列只读 HTTP 端点（与官方商城结构对齐），插件端即可作为 `registry=third-party` 来源接入；下载产物仅落入用户本地，不经 CodePilot 后端。

```
GET  {registryUrl}/v1/packages?type=skill|mcp&q=&page=&size=
GET  {registryUrl}/v1/packages/{slug}
GET  {registryUrl}/v1/packages/{slug}/versions
GET  {registryUrl}/v1/packages/{slug}/versions/{version}/manifest
GET  {registryUrl}/v1/packages/{slug}/versions/{version}/download    # 200 + 二进制；MIME: application/zip
GET  {registryUrl}/.well-known/codePilot-registry.json               # 必需：声明公钥与能力
```

`well-known` 文件示例（必需，注册中心首次接入时由插件端拉取与验签）：

```jsonc
{
  "name": "My Corp Skills",
  "registryVersion": "1.0",
  "publicKey": "-----BEGIN PUBLIC KEY-----\nMIIBIjANBgkqhkiG9w...\n-----END PUBLIC KEY-----",
  "supportedTypes": ["skill","mcp"],
  "capabilities": {
    "search": true,
    "auth":   ["bearer","apikey","mtls"],
    "rateLimit": { "qps": 5 }
  },
  "tos":     "https://example.com/tos",
  "contact": "security@example.com"
}
```

`manifest` 必须满足下述最小子集（其它字段可选；签名字段 `signature` 由 `publicKey` 对应私钥对整个 zip 的 sha256 进行签名）：

```jsonc
{
  "id": "skill.thirdparty.example",
  "version": "0.4.2",
  "type": "skill|mcp",
  "source": "user",                         // 第三方市场强制 user
  "scope":  "project|global",               // 仅作"建议默认值"，最终由用户在 IDE 中选择
  "title":  "Example",
  "license":"MIT",
  "permissionsBrief": { "tools": ["fs.read","fs.replace"] , "risk":["low","medium"] },
  "triggersBrief":    [ {"language":["java"]} ],
  "downloadUrl":      "https://cdn.example.com/dl/skill.thirdparty.example-0.4.2.cps",
  "sha256":           "<hex>",
  "signature":        "<base64>",           // 对 zip 的 sha256 的签名；插件用 publicKey 验签
  "minPluginVersion": "1.4.0",
  "dependencies":     []
}
```

约束（插件端强制）：
- 任意端点不通过 https 一律拒绝（除非 registry 在 Settings 中显式标记为 `allowInsecure=true`，仅限内网调试）。
- `downloadUrl` 域名可与 `registryUrl` 不同，但响应必须满足 `sha256` 与 `signature` 双校验。
- 第三方 manifest **不允许**含 `systemPrompt / examples / tests` 等字段（这些只在客户端解压后从 zip 内读取），从而避免出现"未签名的提示词正文直接在网络上明文传输"的风险。
- 第三方 manifest 中即便给 `source=system` 也会被插件端强制改为 `source=user` 处理；同 id 与 system Skill 冲突时按 §6.4 规则 system 胜出。
- 插件每次安装都把"registry name + publicKey 指纹 + 包 sha256"上报到后端 `/audit/event`，便于事后追责（仅元数据）。

> CodePilot 后端**不**作为第三方包的代理或镜像；下载链路在插件端与第三方 CDN 之间直连，安全由签名机制保证。这同时保证 system Skill 永不下发的边界。

## 8. 会话辅助接口（不存内容，仅辅助）

> 后端不存会话内容；以下接口只做：摘要生成、命题改写、限流配额。

### 8.1 `POST /session/title`
让模型为新会话起一个简短标题。
```json
{ "messages":[ {"role":"user","content":"..."} ] }
```
返回：`{ "data":{ "title":"OrderService 幂等改造" } }`

### 8.2 `POST /session/digest`
对历史消息做摘要，用于断点恢复。后端服务端独立接口，非 Agent 主流程使用。
```json
{ "messages":[ ... ], "keepRecent": 6 }
```
返回：`{ "data":{ "summary":"...", "recent":[...]} }`

> 注：Agent 主流程中的"对话压缩"**不走本接口**，而是在 `/conversation/run` (mode=agent) 的同一轮模型调用里由 Prompt 引导模型输出 `digest` 事件。本接口仅用于插件端在非 Agent 场景下（例如打开旧会话时做一次性摘要）的辅助调用。

## 10.5 插件自更新

### 10.5.1 `GET /plugin/manifest?channel=stable&ideBuild=242.20224&pluginVersion=1.4.2&deviceId=...`
返回当前通道下"应使用的版本"manifest（仅元数据；任何阶段都不会下发 system Skill 正文）。

请求头：`Authorization: Bearer ...`、`X-CodePilot-Device-Id: ...`。

```jsonc
{
  "code":0,
  "data":{
    "version":"1.5.0",
    "channel":"stable",
    "minIdeBuild":"232",
    "maxIdeBuild":"243.*",
    "rolledOut":true,                      // 是否对当前 deviceId 已灰度命中
    "pinTo":null,                          // 非空表示后端要求回滚到指定版本
    "artifacts":[
      {
        "kind":"hot-patch",                // hot-patch | full
        "url":"https://cdn.codePilot.<corp>/dl/codePilot-1.5.0.cpp.zip",
        "size":1532145,
        "sha256":"...",
        "signature":"<base64>",
        "covers":["webui","prompts.cache","tokenizer.vocab","i18n"]   // 描述本包覆盖的资源类
      },
      {
        "kind":"full",
        "url":"https://cdn.codePilot.<corp>/dl/codePilot-1.5.0.zip",
        "size":12834560,
        "sha256":"...",
        "signature":"<base64>"
      }
    ],
    "changelogUrl":"https://docs.codePilot.<corp>/release/1.5.0",
    "checksumAlgo":"SHA-256",
    "ttlSeconds":900
  }
}
```

约束：
- 客户端**优先选择 `kind=hot-patch`**；若 `covers` 内含未在白名单（见插件端 §14.3）中的资源类则降级 `kind=full`。
- `signature` 必须由平台公钥校验通过；不通过即丢弃并审计 `update_signature_invalid`。
- 通道：`stable / beta / dev / corp.<name>`；企业版可强制 `corp.<name>`。

### 10.5.2 `POST /plugin/update/report`
客户端在升级关键节点上报状态（不带 system Skill 正文）。

```json
{
  "phase":"checked|downloaded|verified|applied|deferred_until_restart|failed|rolled_back",
  "from":"1.4.2",
  "to":"1.5.0",
  "kind":"hot-patch|full",
  "durationMs":4231,
  "errorCode":null,
  "fingerprint":"sha256:..."
}
```

返回：`{ "code":0, "data":{} }`。后端用于灰度统计与异常告警，不存任何会话内容。

### 10.5.3 `GET /plugin/changelog/{version}`
返回结构化变更日志（用于 ToolWindow 通知卡片展示）：`{ added: [], changed: [], fixed: [], security: [] }`。
