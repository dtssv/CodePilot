## 3. 上下文采集（ContextCollector）

收集的上下文统一抽象为 `ContextItem`：

```jsonc
{
  "type": "selection|file|symbol|diagnostic|workspace",
  "language": "java",
  "path": "src/main/java/com/x/Foo.java",
  "range": {"startLine": 10, "endLine": 60},
  "content": "...",
  "psiSummary": {           // 可选，由 PSI 提取
    "kind": "method",
    "name": "doRefactor",
    "signature": "public List<String> doRefactor(Foo)",
    "imports": ["java.util.List"]
  },
  "tokensEstimate": 1240
}
```

- 选中代码：使用 `EditorEx.getSelectionModel`。
- 选中文件：从 ProjectView 多选。
- 大文件：>2k 行时只发送选中段 + PSI 摘要 + 文件 outline。
- 工程上下文：通过 `rag-service` 的轻量索引在后端按需检索，不一次性上传整个工程。

## 4. 插件端工具体系（Plugin-Side Tool Suite）

> 插件端**自带一组工具**，由模型通过 ToolCall 调用、由插件本地执行并把结果回传后端。后端在 `/conversation/run` 之前会把这组工具的 Schema 注入 Prompt，模型只能在白名单内选择。
>
> 工具命名空间：`fs.*`（文件）、`shell.*`（命令）、`plan.*`（计划展示）、`ide.*`（IDE 能力）、`mcp.*`（来自已安装 MCP）。

### 4.0 统一调用约定

- 所有工具采用 OpenAI function-calling 风格描述（JSON Schema），与后端 ToolRouter 保持一致。
- 调用时机：模型回复中包含 `toolCall: {name, args, id}`；插件按 `name` 路由到本地实现。
- 结果回传：`POST /v1/conversation/tool-result`，统一结构 `{ok, result|error, durationMs, meta}`。
- 风险等级：`low | medium | high`。`high` 必须由用户在 IDE 内确认（除非命中"自动应用白名单"）。
- 速率/超时：每工具有默认超时与并发上限；可在 Settings 调整。

### 4.1 工具清单

| 名称 | 说明 | 风险 | 执行端 |
|---|---|---|---|

| `fs.list` | 列目录（受工程根 + 黑名单限制） | low | client |
| `fs.create` | 新建文件（含目录自动创建） | medium | client |
| `fs.write` | 整文件覆盖（必经 Diff 预览） | medium | client |
| `fs.replace` | 局部文本/正则替换（首选） | medium | client |
| `fs.delete` | 删除文件/目录（移到 Trash） | high | client |
| `fs.move` | 移动 / 重命名 | high | client |
| `fs.search` | 工程内文本搜索（基于 IDE find-in-path） | low | client |
| `fs.outline` | 给定文件返回 PSI outline（类/方法签名） | low | client |
| `shell.exec` | 执行单条命令（OS 自适配，超时/截断/黑名单） | high | client |

```jsonc
      "ignoreCase":{"type":"boolean","default":false},
```

### 4.3 工具执行通道与回传

```
模型 → SSE event: tool_call {id, name, args, risk}
插件 → 风险确认（high 必须用户确认）→ 本地执行
插件 → POST /v1/conversation/tool-result {toolCallId, ok, result|error, durationMs}
后端 → 把 result 注入下一轮 Prompt → 继续 SSE
```

### 4.4 安全护栏（在执行前由插件强制）

- 路径必须落在 **当前工程根**（`Project#getBasePath`）之内；越界直接拒绝并回 `error.path_out_of_scope`。
- 黑名单：`.git/`、`.idea/`、`node_modules/`、`*.env*`、密钥文件类（`*.pem`,`*.p12`,`*.key`）默认禁写禁删。
- `fs.replace` 的 `expectMatches` 不一致 → 拒绝（防全局误改）。
- `shell.exec` 命令黑名单：`rm -rf /`、`mkfs`、`shutdown`、`format`、`del /f /s /q C:\\` 等；可在 Settings 增白。
- 所有操作走 IDEA `WriteCommandAction` + `UndoManager`，可一键撤销。

### 4.5 FileTool 实现细节

- 全部走 `WriteCommandAction.runWriteCommandAction`，`UndoManager` 可撤销。
- 写操作产出 `Patch`，调用平台 `DiffManager.showDiff` 让用户 Approve / Reject / 部分接受。
- "自动应用白名单"：按路径前缀、扩展名、单次最大改动行数控制是否免确认。
- `fs.move` 走 `RefactoringElementFactory`，自动更新引用。

### 4.6 ShellTool 实现细节（跨平台命令执行）

```kotlin
data class ShellRequest(
  val command: String,
  val cwd: String?,
  val timeoutMs: Long = 60_000,
  val env: Map<String, String> = emptyMap()
)
```

- 使用 IDEA `GeneralCommandLine` + `CapturingProcessHandler`。
- OS 适配：
  - Windows：默认 `powershell.exe -NoProfile -Command`，可切 `cmd /c`。
  - macOS / Linux：`/bin/bash -lc`。
- 黑名单 + 白名单（用户在 Settings 配置）。
- 输出回传 `{exitCode, stdout, stderr, durationMs, truncated}`。
- 失败时弹"自动修复"选项（默认开），把错误回灌到 Agent。

## 6. SessionStore（本地会话）

- 默认目录：`~/.codePilot/sessions/`，可在 Settings 修改。
- 文件格式：NDJSON，便于追加写入与按行恢复。
- 会话索引：在根目录维护 `index.json`（仅元数据），用于会话列表性能。
- 加密：可选 AES-GCM（密钥放 `PasswordSafe`），用于敏感工程。

会话切换 API（插件内部）：
- `SessionService.list()`
- `SessionService.create(meta)`
- `SessionService.append(sessionId, message)`
- `SessionService.checkpoint(sessionId, payload)`
- `SessionService.resume(sessionId)`
- `SessionService.savePlan(sessionId, plan)` / `loadPlan(sessionId, version)`
- `SessionService.saveSummary(sessionId, digest)` / `latestSummary(sessionId)`
- `SessionService.estimateTokens(sessionId)`：基于本地 tokenizer（如 jtokkit）粗估当前上下文规模

## 6.1 Plan 面板 + Task Ledger 面板（Agent 模式专用，纯本地视图）

两个并列面板，**都不直接调用后端**，由 SSE 流里的 `plan / plan_delta / task_ledger` 事件驱动。

### Plan 面板（步骤级）
- 数据来源：`plan` 全量 / `plan_delta` 增量；插件端按版本号合并；落盘 `plans/plan-{n}.json`。
- 显示：步骤树，每步含 `状态徽章 / 风险标签 / 工具图标 / 预期产物 / acceptance 校验项`。
- 状态机：`pending → running → success | failed | skipped | cancelled`。

### Task Ledger 面板（子任务级）
- 数据来源：`task_ledger` 事件；落盘 `ledger.json`。
- 显示：`goal` 顶部展示，下面是 `subtasks[]` 卡片（id/title/why/status/blocker），用 `cursor` 高亮当前聚焦子任务，`notes` 折叠展示。
- 状态机：`pending → in_progress → done | blocked | cancelled`。

### 用户操作（统一进入 `userPlanEdits` 队列，下次 run 作为 Prompt 输入）
- Approve / Skip / Reorder / Edit 单步；
- Pin 一个子任务为 `cursor`；
- 在子任务卡上"追加备注"或"标记 blocker"；
- 强制重新规划（下一次请求带 `policy.replanHint=true`）；
- "继续"按钮：以 `intent="continue"` 发起下一次 `/conversation/run`；
- "回答"输入框：当上一轮 `done.reason="awaiting_user_input"` 时启用，提交以 `intent="answer"`。

### 风险策略
- `high` 步骤在执行前必须人工确认（结合 `risk_notice` 卡片展示影响预览 + 缓解项）；
- `medium` 默认确认，可白名单免确认；`low` 自动放行；
- 危险命令、跨工程路径、大批量删除会自动升级为 `high`。

## 6.2 上下文工程（MNI + 分层窗口）

为了"既不超长、又不丢关键信息"，客户端采用"分层上下文 + 最小必要信息（MNI）"策略：

### 分层窗口（每次请求的组装来源）

| 层 | 数据来源 | 是否必带 | 构造方式 |
|---|---|---|---|
| `pinned` | 用户显式 pin + 模型 `hintsForContext.pin` | 是 | 保留 path+range+sha1+原文或摘要 |
| `taskLedger`（极简） | 本地 `ledger.json` | 是 | 仅 goal/cursor/当前分支子任务+notes |
| `lastPlanDigest` | 本地 `plans/latest.json` | 是 | 仅 cursor 所在及其依赖步骤 + 已完成步骤 id 列表 |
| `lastAssistantTurnSummary` | 模型上轮 `summaryForNextTurn` 或兜底摘要 | 是 | ≤400 tokens |
| `sessionDigest` | 本地 `summaries/latest.json` | 若存在 | 直接使用 |
| `contexts.recent` | `messages.ndjson` 末尾 M 条 | 是 | FIFO；M 由设置决定（默认 6，压缩后 ≤2） |
| `contexts.refs` | 插件 PSI 索引/历史读过的文件 | 可选 | 仅 path+range+sha1，不送原文 |
| `completedToolCallsTail` + `earlierToolCallsCount` | `steps.ndjson` | 是 | 末尾 K=5 条完整，早期仅计数 |

### 发送前本地预算（`SessionService.prepareRequest`）

```kotlin
fun prepareRequest(...): ConversationRunRequest {
  val budget = settings.contextBudgetTokens                 // 默认 24k
  val mustKeep = pinned + ledgerCursorBranch + lastPlanDigest + summary
  var body = mustKeep + recentTop(6) + refs(outlineOnly=true) + digestIfAny + tailToolCalls(5)
  while (estimate(body) > budget) {
    if (body.recent.size > 2) body = body.shrinkRecent()
    else if (!body.hasLocalDigest) body = body.foldIntoLocalDigest()
    else { body = body.withCompactRequest(); break }
  }
  return body
}
```

### 响应后本地持久化
- 收到 `task_ledger` → 更新 `ledger.json`；
- 收到 `plan / plan_delta` → 合并到 `plans/plan-{n}.json`；
- 收到 `self_check` / `final` 中的 `summaryForNextTurn` → 覆写本地 `lastAssistantTurnSummary`；
- 收到 `self_check.hintsForContext.pin` / `unpin` → 更新 pinned 清单；
- 收到 `tool_result_ack` 后：在本地 `steps.ndjson` 记录 toolCallId + 200 字 summary；
- 收到 `digest` → 覆写 `summaries/latest.json`，把已被 boundary 覆盖的历史从 `messages.ndjson` 移到 `messages.archived.ndjson`。

### 何时触发服务端压缩
- 本地估算 `body > 0.7 * modelContextWindow`；或消息 > 80；或 Agent 步骤 > 12；或用户手动。
- 仅发送 `policy.requestCompact=true`，不需要别的接口；模型在同一次回复先产出 `digest` 再继续。

## 6.3 提问与回答交互（needs_input / answers）

当 SSE 收到 `needs_input`（由 Prompt §9.7 产出）：

### UI 展现
- 输入区上方弹出"**需要你确认几件事**"卡片，包含 `title / reason / notesForUser`。
- 每个问题渲染为一张卡片：
  - 标题行显示编号（index）与 `prompt`；
  - 下方展示 `options` 为可点选的 Chip（带 `impact` 徽章与 pros/cons tooltip）；
  - `defaultOptionId` 对应 Chip 高亮为"推荐"；
  - `kind="freeform"` 时显示文本框 + `placeholder`；
  - 底部行提供"输入编号+答案"文本框（占位符："例如：1: b; 2: a; 3: 600"）与"自由输入"按钮（切到普通输入框）。
- 底部按钮：`发送（n/N 已回答）` / `跳过` / `取消任务`。

### 三种回答方式（统一打包为 `answers[]`）
1) **点选** → `{questionId, optionId}`；
2) **编号+答案** → 正则 `^\s*q?(\d+)\s*[:：\-\s]\s*([a-zA-Z0-9][\w\-]*)\s*# 02 — IDEA 插件端详细设计


## 7. 断点恢复流程

```
启动会话 → 每条 user/assistant/tool 消息追加 messages.ndjson
Agent 每完成一步 → 追加 steps.ndjson + 写 checkpoints/{n}.json
异常中断/手动 Stop → 标记 last step 为 INTERRUPTED

恢复：
  1) 加载最后一个 checkpoint
  2) 调用 POST /conversation/resume，body={sessionDigest, lastCheckpoint, plan}
  3) 后端跳过已完成步骤的副作用，仅基于摘要续推
```

`sessionDigest` 由插件端生成：把 messages 做摘要（保留最近 N 条原文 + 早期摘要），避免越发越长。

## 8. 模型配置

Settings → CodePilot → Models：
- **内置模型**：后端下发清单（如 `codePilot-default`, `codePilot-pro`），不暴露 ApiKey。
- **自定义服务商**：
  - 字段：`name, baseUrl, apiKey, model, protocol(openai/azure-openai/ollama/anthropic), headers, timeout`。
  - 校验：调用 `POST /models/test` 走后端校验，避免插件端直接持有任意服务商 Key 时仍可被审计。
  - 存储：ApiKey 放 `PasswordSafe`；其余放 `PersistentStateComponent`。

## 9. MCP / Skill 商城与本地管理

> **重要原则**：system Skill / MCP **只在后端**，任何情况下都不会下载到用户本地；商城分发链路**只服务 user 域**。用户域产物由用户选择作用域后落到对应目录（见 §9.0）。

ToolWindow → Marketplace：
- Tab：`MCP Servers` / `Skills`。
- 列表：分类、热度、版本、签名校验状态、`triggers` 摘要、`source` 标记（system / user）。system 条目只读，不提供"下载"按钮；user 条目可安装。
- 详情：README、权限声明（文件读/写/网络/命令、`permissions.tools`、`risk` 上限）、变更日志、`tests` 通过情况、`audit.tokensEstimate`。
- 操作：
  - **Install to project**（作用域 `project`，落 `<projectRoot>/.codePilot/...`）
  - **Install to global**（作用域 `global`，落系统配置目录，见 §9.0）
  - 卸载 / 更新 / 启用 / 禁用 / 配置环境变量
- 作用域二选一，重复 id 冲突 `project > global`；安装完成后记录到该作用域下的 `index.json`。

### 9.0 作用域与本地路径

> **user 域产物路径（二选一安装）**，**system 域永远不落本地**：

**project（项目生效）**：
```
<projectRoot>/.codePilot/
├── skills/installed/<id>@<version>/skill.yaml
├── skills/index.json      # scope=project
├── skills/disabled.json
└── mcps/...
```

**global（全局生效，按 OS 遵循本地配置约定）**：
- macOS：`~/Library/Application Support/CodePilot/{skills,mcps}/...`
- Linux（XDG）：`~/.config/codePilot/{skills,mcps}/...`
- Windows：`%APPDATA%\CodePilot\{skills,mcps}\...`

切换工作区时，仅 project 作用域随当前 `projectRoot` 变化；global 保持不变。

### 9.1 Skill Tab 专属
- 列表筛选：按 `language / framework / action / scope（project/global/system）/ source`；
- 详情可预览生效条件（`triggers`）与示例（few-shot，仅 user Skill 提供正文；system 只看安全摘要）；
- 用户可在"Skill 设置"里：
  - 钉版本（`skills.requested: ["skill.lang.java@1.2"]`）；
  - 全局禁用（`skills.disabled: ["skill.scenario.read-conventions-first"]`）；
  - 切换同一 id 的 project/global 覆盖关系（仅限 user Skill）；
  - 调整 `priority` 覆盖默认排序（仅限自定义 Skill）；
- 与请求联动：每次 `/conversation/run` 自动把"本轮可能命中的 user Skill"打包到 `userSkills[]` 上行（接口 §3.1.2），每项带 `scope: project|global`；同时注入 `skills.requested / disabled`；
- 安全：第三方 Skill 仅声明性 prompt + yaml；不允许包含可执行代码；权限声明的工具会在 ToolRouter 处再校验一次；user Skill 严禁 `merge: override` 任何 system Skill。
- 本地索引：`index.json` 记录 `id, version, scope, source, sha256, signedBy, installedAt`。

### 9.2 MCP Tab
- 本地 MCP：通过 stdio 拉起子进程，stdin/stdout 走 JSON-RPC（MCP 协议）。
- 远端 MCP：HTTP/SSE，由后端 MCP Hub 代理；其元数据由用户在本地 `index.json` 维护。
- 安装同样分 `project / global` 作用域；每次 `/conversation/run` 时，插件把当前已启用 MCP 的"工具白名单 + 权限声明"打包为 `userMcps[]` 上行（接口 §3.1.2），后端据此严格校验。

### 9.3 创建自定义 user Skill（向导）
- 向导流程：
  1) 用户填 `id / version / triggers / permissions / priority / merge`；
  2) 在编辑器内编写 `systemPrompt`（英文）+ 可选 few-shot 示例；
  3) `source` 自动填为 `user`；尝试填 `system` 被 IDE 校验拒绝；
  4) **选择作用域：`project` 或 `global`**；
  5) 保存到对应目录（见 §9.0），自动更新 `index.json` 并签名（可选）。
- 自定义 Skill 仅本地生效，不会上传后端；如需对外分享，用户可手动导出 zip 并通过商城上架（经签名审核后即变 system，由后端托管）。

### 9.4 作用域冲突与推荐规则
- 同一 `id` 在 project 与 global 同时存在：`project` 胜出；列表中高亮"被 project 覆盖"。
- 推荐：
  - **project** 适合"这个工程专属的规范 / 私域规则"（可随 git 共享）；
  - **global** 适合"跨工程的团队偏好 / 个人习惯"（不随工程迁移）。
- 强制企业下发的规范不在此两类里——企业规范应由官方渠道以 system Skill 形式推送到后端，不下发到本地。

### 9.5 安装来源总览（四种渠道）

> 适用对象：**user Skill / MCP**。**system Skill / MCP 不可被安装**，UI 明确屏蔽"下载"按钮。

| 来源 `registry` | 信任级别 | UI 入口 | 说明 |
|---|---|---|---|
| `official`（官方商城） | high | Marketplace Tab | 对接后端 `GET /v1/mcp/packages`，平台签名；默认启用 |
| `third-party`（外部市场） | medium（"已信任"后）/untrusted | Settings → Registries | 用户添加外部 Registry URL 与公钥，满足 §7.9 协议；未信任前每次安装都要确认 |
| `local`（本地文件） | low | ToolWindow 顶部 ➕ / 拖拽 | 导入本地 `*.cps` / `*.mcp` 包 |
| `builtin-ide`（IDE 新建向导） | low | Marketplace Tab 右上角"+" | 直接在本地新建 `skill.yaml` / `mcp.yaml`，无需下载 |

**无论来源如何，都在"Install"按钮上清晰出现两个选项**：`Install to project` / `Install to global`。

### 9.6 从"官方商城"安装（registry=official）

UI：Marketplace Tab 里选择条目 → 右下角两个主按钮 `Install to project` / `Install to global`。

流程：

```
[UI 点击 Install to <scope>]
  → PackageInstaller.resolve(registry=official, slug, version)         # GET /v1/mcp/packages/{slug}/versions/{version}/manifest
  → preflight(manifest)                                                 # 权限/依赖/OS 兼容检查
  → download(manifest.downloadUrl)                                      # HTTPS，签名 + sha256 双校验
  → confirmDialog(permissions, triggersBrief, signedBy, license)        # 用户必须勾选"我理解并允许"
  → extractTo(<scope> 目录下 installed/<id>@<version>/)
  → indexRebuild(<scope>)
  → audit.reportInstalled(registry=official, id, version, scope)        # 仅元数据
```

取消条件：签名失败、权限超白名单、依赖缺失、用户拒绝 → 立即终止并回滚目录。

### 9.7 从"第三方市场"安装（registry=third-party）

**前置**：Settings → CodePilot → Marketplace Registries 中添加外部市场，填入：

| 字段 | 说明 |
|---|---|
| `name` | 市场显示名（如 `My-Corp-Skills`） |
| `registryUrl` | Registry 根 URL，须满足 §7.9 协议 |
| `trustedPublicKey` | 该市场的签名公钥（PEM） |
| `auth` | 可选：Bearer Token / API Key / mTLS 证书 |
| `enabled` | 开关 |

UI：添加后，Marketplace Tab 顶部出现"来源筛选器"，可切换到该市场浏览。

流程：

```
[UI 点击 Install to <scope>]
  → GET {registryUrl}/packages/{slug}/versions/{version}/manifest       # 结构需符合 §7.9
  → verify signature with trustedPublicKey                              # 未通过 → 阻断
  → preflight(manifest)
  → download(manifest.downloadUrl)  + sha256 校验
  → 额外提醒："此包来自外部市场 <name>，权限如下 ... 是否信任并安装？"
     - 若第一次从该 registry 安装任意包，要求用户勾选 "信任此 registry 的签名后续自动校验"
  → confirmDialog(...)
  → extractTo / indexRebuild / audit.report(registry=third-party.<name>)
```

失败路径：
- 未找到 `trustedPublicKey` → 阻止安装并提示补全；
- 签名主体与 `trustedPublicKey` 不匹配 → 一律阻止，不提供"强行安装"。

离线镜像：企业可以部署自己的 third-party registry 作为代理；插件端 UX 与上面一致，仅 `registryUrl` 指向内部。

### 9.8 从"本地文件"安装（registry=local）

使用场景：团队私域离线共享、用户导出再导入、测试调试。

包格式（两种等价形式）：
- `*.cps`（Skill 包）：zip 结构
  ```
  skill.yaml
  README.md              (可选)
  LICENSE                (可选)
  examples/*             (可选 few-shot)
  tests/*                (可选回归用例)
  signature.bin          (可选；若来自官方/第三方签名)
  ```
- `*.mcp`（MCP 包）：zip 结构
  ```
  mcp.yaml               (runtime / entry / permissions)
  bin/                   (可选可执行 / Node 入口脚本)
  README.md              (可选)
  signature.bin          (可选)
  ```

UI 入口：
- ToolWindow → Marketplace Tab → 顶部"➕ Install from file…"；
- 也支持**把文件拖拽到 Marketplace Tab**。

流程：

```
[用户选择 file.cps 或 file.mcp]
  → 解析 manifest；如有 signature.bin 尝试按已信任公钥列表校验
  → 未签名 / 签名未通过 → 弹更严格的确认：
     "此包未通过签名校验，风险较高，仅建议在你信任的来源下使用。是否继续？"
     用户勾选"我明白风险"后才可继续
  → preflight(manifest)
  → confirmDialog(permissions, triggersBrief)
  → scope-pick：project / global
  → extractTo / indexRebuild / audit.report(registry=local)
```

### 9.9 通过"IDE 新建向导"直接创建（registry=builtin-ide）

用于"我只想为当前工程写个规范，不必经过任何市场"。

UI：Marketplace Tab → 右上角 "New Skill" / "New MCP"。

向导步骤（以 Skill 为例）：

```
Step 1  基本信息：id / version / title / description / license
Step 2  生效条件 triggers（language / framework / fileGlob / keywords）
Step 3  输出声明 outputs（plan.*, patches, selfCheck.*...）
Step 4  权限 permissions.tools（勾选当前白名单里的子集）
Step 5  Prompt 编辑器（英文，≤500 tokens；live token 估算；JSON Schema 字段校验）
Step 6  可选 few-shot 示例（按 {{userLocale}} 写）
Step 7  作用域 scope：project 或 global
Step 8  Review → 创建
```

创建后：
- 文件落到对应作用域的 `installed/<id>@<version>/skill.yaml`；
- 自动在 `index.json` 注册（`source=user`, `signed=false`）；
- 可"立即测试"：在 Playground 侧栏触发一个单轮请求，仅激活该 Skill，观察 SSE `skills_activated` 是否包含它、输出是否符合 Prompt §9.6 selfCheck。
- 可"导出为 `*.cps`"，便于分享或上架到商城；导出时要求用户选择是否用本地密钥签名。

MCP 新建向导同理，但 Step 5 由 Prompt 编辑器变为"运行时声明"（Node/Python/Java 入口 + stdio/http 模式 + 工具 Schema 声明），Step 6 可选择"联调一次"以验证 MCP 子进程可正确响应 `tools/list`。

### 9.10 卸载、启用/禁用、更新与导出

- **卸载**：右键条目 → Uninstall；默认走"软卸载"（移入 `disabled.json`，保留文件 14 天）；勾选"彻底删除"则移除目录。
- **启用/禁用**：不影响文件，只切换 `disabled.json`。
- **更新**：
  - 官方 / 第三方：轮询 manifest 的最新版本；用户手动或"自动更新开关"触发；更新流程 = 下载新版 → 并存 → 切换指向 → 删除旧版；
  - 本地文件：不自动更新，提示"请手动覆盖安装"；
  - 新建向导：每次修改 Prompt 即增 patch 版本号；版本间可在"版本历史"里回滚。
- **导出**：右键 → Export → 选择是否签名 → 生成 `*.cps` / `*.mcp`。不允许导出 system 条目。

## 10. 安全与隐私

- 所有出网请求只到 `https://api.codePilot.<corp>`，TLS 1.2+，Pinning 可选。
- 请求签名：`X-CodePilot-Signature = HMAC_SHA256(body + ts + nonce, deviceSecret)`。
- 重放保护：服务端校验 `ts` ±5min，`nonce` 入 Redis 5min 去重。
- 敏感目录黑名单（默认）：用户 Home 之外目录、`.git/objects`、`node_modules`、`.idea` 内的私密文件等不会被自动扫描。
- 上行内容脱敏：邮箱/手机号/身份证/AK-SK 在插件端正则脱敏，后端二次脱敏。
- 日志：插件端 INFO 不打印代码内容；DEBUG 仅在 `idea.log` 本地。

## 11. 关键交互时序（以"重构选中代码"为例）

```
User 选中代码 → 右键 Refactor → 插件 ContextCollector 收集 ContextItem
插件 → POST /v1/actions/refactor (SSE)
后端 → ChatClient.stream(prompt)
后端 → SSE: token / patch / done
插件 → 流式渲染 + 收到 patch 后调 DiffManager 预览
User → Approve → FileTool.replace 应用 → 写 messages.ndjson
```

## 12. Agent 模式时序（单次调用，Plan→Compact?→Execute 一次完成）

> Chat 与 Agent **同一个输入框、同一个 SSE 端点 `/v1/conversation/run`**，仅由请求体里的 `mode` 字段切换行为；UI 上的模式开关只是把 `mode` 从 `chat` 切到 `agent`，并附带 `tools / userPlanEdits / policy.selfCheck` 等 agent 专属字段。

```
User 输入复杂任务 + 模式 = agent
插件 ContextCollector 收集上下文，本地估算 tokens
插件 → POST /v1/conversation/run { mode:"agent", policy.requestCompact:"auto", policy.selfCheck:true, userPlanEdits, contexts, tools, ... } (SSE)

后端 PromptOrchestrator 拼装 system：
  base + agent + tools + (可选) compact + (可选) replan + (可选) userEdits + (turn>=2) selfCheck + guard

模型回复（同一次调用，可流式分段）：
  (可选) digest{...}                 // 命中压缩条件时先吐
  plan{...} or planDelta{...}
  (turn>=2) selfCheck{...}           // 上一轮工具调用后必填
  toolCall{name,args,risk}           // 第一步动作
  done

插件：
  - 收到 digest → 折叠历史 → 落 summaries/
  - 收到 plan/planDelta → 合并到 Plan 面板
  - 收到 selfCheck → 在对应工具卡片上展示自检结论/证据；高亮 nextAction
  - 收到 toolCall → 风险确认 → 本地执行 → POST /v1/conversation/tool-result

后端拿到 result → 触发下一轮模型调用（仍在同一 SSE 通道里）：
  模型先吐 selfCheck（对刚才的 toolCall）→ 决定 continue|retry|replan|finalize → 下一个 toolCall 或 final
  ...直到 final 或 max_steps / stopped

mode = chat 时：
  - 不发送 tools，不出现 plan/planDelta/selfCheck/tool_call；
  - SSE 只产生 (digest?) → delta* → usage → done。

单次 run 推进一段，命中自然停顿（subtask_done / awaiting_user_input / max_steps）后 done。
之后插件根据用户操作发起下一次 run（intent=continue|answer），把 lastPlan/taskLedger/completedToolCalls 传回，
模型基于已有进度继续推进，直到 goal 达成。
```

## 13. 多次 run 协作时序（"先读规范再动手"为例）

```
Run 1  insight=new
  user: "为 OrderService 增加幂等控制并补充测试"
  → SSE: task_ledger(goal,subtasks[t1..t4],cursor=t1)
        plan(全量, s1=读 docs/coding-style.md, s2=读 OrderService.java, s3=方案草案, s4=补丁, s5=跑测)
        tool_call(fs.read("docs/coding-style.md"))
  插件 → POST /v1/conversation/tool-result(...)
  → SSE: tool_result_ack
        self_check(ok=true, matchedExpectation=true, evidence: 引用规范关键条款)
        plan_delta(markStatus s1=success, 更新 s3 输入)
        task_ledger(notes 追加: "Java 17, JUnit 5, Spotless 启用", cursor=t1 done → t2)
        final.subtaskDone=true, answer="已读取规范，准备进入梳理 Service 阶段"
        done.reason=subtask_done, continuationToken=ctk-1

Run 2  intent=continue, lastPlan/taskLedger/completedToolCalls 回传
  → SSE: plan_delta(...)
        tool_call(fs.outline("OrderService.java"))
        ...
        needs_input(q1: "公共签名是否允许变更？") → done.reason=awaiting_user_input

Run 3  intent=answer, input="不允许变更签名"
  → SSE: plan_delta(更新 s3 方案=包装方法)
        tool_call(fs.replace 添加幂等校验)

  插件用户 Approve → 应用补丁 → tool-result
```

## 14. 插件自更新（热更新优先 + 重启更新兜底）

> 目标：CodePilot 插件不依赖 IDE 应用商店的发布节奏，由后端控制版本通道与灰度；优先**热更新**（不重启 IDE），不可热更时回退**重启更新**（关闭 IDE → 旧文件被替换 → 重新打开）。

### 14.1 体系总览

```
后端 release-channel
  └─ /v1/plugin/manifest          // 客户端轮询：获得当前通道下的最新版本与 manifest（仅元数据）
       ├─ artifacts:
       │    ├─ kind: hot-patch    // *.cpp.zip：热更新包，仅含可热替换资源
       │    └─ kind: full         // *.zip / *.dmg.tar：完整插件（用于重启更新）
       ├─ minIdeBuild
       ├─ checksum: sha256
       └─ signature: <base64>     // 平台 RSA 私钥签名
```

### 14.2 通道 / 灰度 / 回滚

- **通道（channel）**：`stable / beta / dev / corp.<name>`，由用户在 Settings 选择，企业版可强制 `corp.<name>`。
- **灰度**：后端按 `userId / deviceId / tenantId` 哈希命中比例下发 manifest；命中外仍返回旧版本。
- **回滚**：后端可下发 `pinTo: <version>`，客户端立即回滚至该版本（热更新 → 切换 active；重启更新 → 安排下次重启替换）。
- **签名/校验**：客户端只信任平台公钥校验通过的 manifest 与 artifact；任何校验失败一律拒绝并审计 `update_signature_invalid`。

### 14.3 可热更新对象（白名单）

只有以下"无类装载副作用"的资源才走热更新；其它一律走重启更新：

| 资源 | 是否可热更 | 备注 |
|---|---|---|
| 提示词模板 / Skill 安全摘要缓存 / 配置 (`*.json/*.yaml`) | ✅ | 直接替换文件并 `reload()` |
| ToolWindow JCEF 前端资源 (`webui/*.js/*.css/*.html`) | ✅ | JCEF 重新加载页面 |
| Token / Tokenizer 词表 | ✅ | 卸载旧实例 + 加载新实例 |
| 文案 / i18n 资源包 | ✅ | 触发 `LocalizationManager.reload()` |
| 业务编排（Kotlin）类、Action、Intention | ❌ | 涉及 IntelliJ ClassLoader，不可热替换 |
| native lib / JCEF 二进制 / IDEA SDK 升级 | ❌ | 必须重启 |
| 任何与 `plugin.xml` 注册项相关的变更 | ❌ | IntelliJ 限制 |

> 如果 manifest 含有 `kind=hot-patch` 但实际涵盖了 ❌ 类资源，客户端拒绝热更并降级为重启更新，并审计 `hot_patch_unsafe`。

### 14.4 升级流程

```
状态机：Idle → Checking → Downloaded → Verified → Applied | DeferredUntilRestart | Failed → Idle
```

1. **检查**：启动后 + 每 N 分钟 + 用户手动触发 → `GET /v1/plugin/manifest`。
2. **下载**：仅当当前版本与 manifest 不一致；下载到 `~/.codePilot/updates/staging/<version>/`。
3. **校验**：sha256 + 签名；不通过即丢弃并报告。
4. **应用**：
   - **热更新**：使用单写多读的 `RuntimeResourceRegistry` 切换至新资源；旧资源延迟回收；切换原子化（per-component CAS）；任何失败立刻回滚到旧资源；
   - **重启更新**：把 `staging/<version>/` 写入 IDEA 的 `pluginsTemp/`，IDEA 关闭时由 `PluginInstaller` 替换；用户得到通知"下次启动生效"。
5. **失败**：清理 staging；上报 `audit.update_failed`；保持当前版本可用，不影响业务。

### 14.5 与 IntelliJ Plugin Update 协作

- 我们**不依赖** IDE 内置 marketplace；但兼容它：用户也可手动安装离线包。
- `plugin.xml` 中声明 `change-notes` 与 `since-build/until-build` 由 manifest 同步生成，便于内置 Plugin 列表展示版本一致。
- 若用户更换通道（dev→stable），客户端在下次拉取 manifest 时会自动按"目标通道"做整版替换（必要时走重启更新）。

### 14.6 客户端 API（内部）

| 入口 | 行为 |
|---|---|
| Settings → CodePilot → Update | 显示当前版本/通道/灰度命中、手动检查、查看变更日志 |
| ToolWindow → 通知卡片 | "新版本可用，立即热更新 / 安排重启更新" |
| Action `CodePilot.CheckForUpdates` | 立即检查 |
| Action `CodePilot.ApplyHotPatchNow` | 立即应用已下载的热更新 |
| Action `CodePilot.RollbackToPreviousVersion` | 回滚到上一已知良好版本 |

### 14.7 安全约束

- 客户端永远不下载并落地"system Skill 正文"——热更新只换"插件自身资源"，不掺 system Skill 内容。
- 升级/回滚不会改动用户会话目录（`~/.codePilot/sessions/`）与 user Skill / MCP（`<projectRoot>/.codePilot/{skills,mcps}/` 或系统配置目录下的 `CodePilot/{skills,mcps}/`）。
- 升级前后插件指纹（`pluginFingerprint`）在 `/v1/audit/event` 中报告，便于排障。

## 15. 强制状态重置（Force Reset & Clean Restart）

> 目标：当插件因损坏的本地状态、僵死的会话、卡住的 SSE 连接、损坏的缓存等导致"重启 IDE 仍卡死"时，提供一条"必然奏效"的恢复路径——通过 IDE 命令面板执行重置，**绕过本地持久化**，让插件以干净状态重启或继续运行。

### 15.1 命令清单（Action / 命令面板可用）

| Action ID | 行为 | 是否需要重启 IDE |
|---|---|---|
| `CodePilot.Reset.Soft`        | 关闭所有进行中的 SSE 流、清空内存中的 Plan/Ledger 缓冲、不动磁盘 | 否 |
| `CodePilot.Reset.SessionsCache`| 清理 `~/.codePilot/sessions/<workspaceHash>/.cache/` 与 IDE 内存状态；保留消息原文 | 否 |
| `CodePilot.Reset.Skills`      | 清理 `~/.codePilot/skills/index.json` 重建索引；不删 Skill 文件 | 否 |
| `CodePilot.Reset.Mcps`        | 杀掉所有由插件拉起的 MCP 子进程并重建索引 | 否 |
| `CodePilot.Reset.Plugin`      | 等价于"插件级 Soft Reset"：禁用 + 启用插件，重建所有插件托管对象 | 否 |
| `CodePilot.Reset.HardLocal`   | 把整个 `~/.codePilot/` **重命名为** `~/.codePilot.broken-<ts>/` 并新建空目录 | 是（自动触发干净重启，见 §15.3） |
| `CodePilot.Reset.Factory`     | `HardLocal` + 清除 PasswordSafe 中本插件保存的所有凭据 + 触发干净重启 | 是 |

> 命名约定：所有 Action ID 以 `CodePilot.Reset.*` 开头；命令面板（`Find Action`，mac：`⇧⌘A`）输入 `CodePilot reset` 即可全部找到。

### 15.2 干净重启（Clean Restart）机制

干净重启的关键在于"**不依赖损坏的本地状态**"也能再次启动；步骤：

1. **写入 sentinel**：把 `~/.codePilot/.flags/clean_start` 文件设为存在（内含 `requestedAt / requestedBy / reason`）。
2. **重命名/清空**：把可疑目录改名（例如 `sessions/ → sessions.broken-<ts>/`、`plans/ → plans.broken-<ts>/`），保留原始数据便于排查。
3. **触发重启**：调用 `ApplicationManager#restart`；若失败（IDE 被冻结）则提示用户"请关闭并重新打开 IDE"。
4. **重启后**：插件 `StartupActivity` 检测到 sentinel：
   - **不读取**任何旧目录；以**内存默认值**启动；
   - 显示醒目通知"已干净重启；旧数据已备份到 `*.broken-<ts>`，可用 'Show in Files' 查看"；
   - 删除 sentinel；
5. **自检**：启动后立即跑 `HealthCheck`：网络 / Token / 工具白名单 / Pong 后端 / Skill 安全摘要拉取；任一失败给出明确修复指引。

### 15.3 命令行/外部触发（IDE 已冻结时也可用）

> 当 IDE 完全无响应、无法打开命令面板时，用户可在系统终端运行下列命令；下次启动 IDE 时，插件 `StartupActivity` 会执行对应重置后再加载。

```
# 软重置（清缓存与索引）
$ touch ~/.codePilot/.flags/reset_soft

# 清空会话缓存（保留消息原文）
$ touch ~/.codePilot/.flags/reset_sessions_cache

# 强力本地重置：把可疑目录改名后干净启动
$ touch ~/.codePilot/.flags/reset_hard_local

# 工厂重置：HardLocal + 清凭据
$ touch ~/.codePilot/.flags/reset_factory
```

- 也提供一条统一脚本 `~/.codePilot/bin/codePilot-reset`，在安装时由插件首启释放，命令式封装上述 sentinel；
- Windows 下脚本同名 `.cmd` 版本，行为一致。

### 15.4 与 IDE 内置缓存清理的关系

- 不替代 IntelliJ `Invalidate Caches`，但 `CodePilot.Reset.Factory` 在执行后会"提示用户也跑一次 IDE 的 Invalidate Caches"以彻底排障；
- 不删除工程文件、不动 VCS 工作区、不修改 `.idea/`（除非确实由插件托管）；删除前一律弹确认。

### 15.5 安全与回退

- 所有"重命名而非删除"的操作给用户兜底，14 天后才允许"清空 .broken-* 目录"。
- 重置不会清空"已安装的 user Skill / MCP"——除非用户在确认对话框里明确勾选；
- 重置不影响后端会话审计（后端本就不存会话内容）。

### 15.6 时序示意

```
User: 命令面板 → CodePilot.Reset.HardLocal
插件:
  1) 写入 sentinel ~/.codePilot/.flags/clean_start  (reason=hard_local)
  2) rename sessions/ → sessions.broken-<ts>/
  3) rename plans/ → plans.broken-<ts>/
  4) ApplicationManager#restart
IDE 重启 → StartupActivity:
  a) 检测 sentinel
  b) 跳过旧目录加载，使用内存默认状态
  c) 提示"已干净重启"，删除 sentinel
  d) 触发 HealthCheck
```
