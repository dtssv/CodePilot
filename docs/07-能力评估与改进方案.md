### 4.2 完全未实现的能力方案

#### 4.2.1 Bug Finder（后台自动扫描）

**设计方案：**

```
模块: plugin/src/.../bugfinder/
├── BugScanService.kt         # 调度器：文件保存后 30s debounce
├── DiagnosticCollector.kt    # 收集 IDEA Inspections 结果
├── BugScanAction.kt          # 右键手动触发
└── BugInlayProvider.kt       # 结果展示：Inlay Hint + Gutter Icon

触发方式:
- 手动: 右键 → CodePilot → Find Bugs
- 自动: 文件保存 debounce 30s（Settings 可开关）

流程:
1. DiagnosticCollector 收集当前文件的 Inspections 结果
2. 把 diagnostics + 相关代码上下文发送后端 POST /v1/actions/bug-scan
3. LLM 返回结构化 Bug 报告
4. BugInlayProvider 在编辑器中渲染 Inlay Hint
5. 点击可触发 Agent 模式修复

接口:
POST /v1/actions/bug-scan (SSE)
  body: { sessionId, files: [{ path, content, diagnostics[] }] }
  events: delta → { bugs: [{ severity, location, description, suggestion, confidence }] }

WebUI 展示:
- 在 ChatView 中以 BugCard 形式渲染
- 每个 Bug 带 "修复" 按钮 → 切换到 Agent 模式
```

#### 4.2.2 Smart Apply（智能定位应用）

**设计方案：**

```
问题: 当模型输出的 patch 的 search 文本因缩进/空格/微小差异无法精确匹配时，
      当前实现直接报错，而 Cursor 能智能找到最佳匹配位置。

方案:
1. PatchApplier.smartSearch(path, searchText, tolerance):
   - 精确匹配 → 直接应用
   - 空白容忍匹配 → 忽略行首缩进差异
   - 模糊匹配 → Levenshtein 距离 ≤ 20% 的最佳位置
   - 行偏移匹配 → 在 ±50 行范围内搜索
   - 全部失败 → 报错回传 Agent

2. 实现策略:
   ```kotlin
   class SmartMatcher {
       fun findBestMatch(fileContent: String, searchText: String): MatchResult? {
           // Phase 1: 精确匹配
           val exact = fileContent.indexOf(searchText)
           if (exact >= 0) return MatchResult(exact, 1.0)
           
           // Phase 2: 空白归一化匹配
           val normalized = normalizeWhitespace(searchText)
           val normFile = normalizeWhitespace(fileContent)
           // ...
           
           // Phase 3: 逐行 Levenshtein
           // 找到编辑距离最小的连续行段
           
           // Phase 4: 按行内容 hash 做滑动窗口
       }
   }
   ```

3. 在 fs.replace 的 ToolDispatcher 中调用:
   若 expectMatches 校验失败但 SmartMatcher 找到唯一最佳匹配 → 自动使用
   若有多个候选 → 发 needs_input 让用户选
```

#### 4.2.3 Privacy Mode（隐私模式 UI 开关）

**设计方案：**

```
Settings → CodePilot → Privacy:
  [x] 零数据保留模式
      - 后端不缓存任何 Redis 数据
      - 不上报审计事件
      - 不发送 telemetry
      - 关闭 RAG 索引上传（纯本地索引本已无上传）
      
  [x] 本地会话加密
      - 使用 AES-GCM 加密 messages.ndjson
      - 密钥存 PasswordSafe
      
  [x] 请求不携带 deviceId
      - 匿名模式，仅用 JWT 鉴权

实现:
- CodePilotSettings 增加 privacyMode: Boolean
- HttpClientService 在 privacyMode=true 时:
  - 不发送 X-CodePilot-Device-Id
  - 不发送 X-CodePilot-Nonce / Signature
  - Header 增加 X-CodePilot-Privacy: strict
- 后端收到 X-CodePilot-Privacy: strict 时:
  - Redis TTL 设为 0（不缓存）
  - 不写审计日志
  - 响应头 X-CodePilot-Data-Retained: none
```

#### 4.2.5 Notepads（上下文暂存区）

**设计方案：**

```
概念: 用户可以创建"笔记本"来持久化常用的上下文片段，
      如架构说明、API 格式要求、常用代码模板等。
      Notepads 的内容会作为 pinned context 每次自动注入。

存储: ~/.codePilot/notepads/<name>.md

UI:
- ToolWindow → Notepads Tab
- 列表展示所有笔记本
- 新建/编辑/删除
- 每个笔记本可标记为 "auto-pin"（每次请求自动注入）

与请求集成:
- auto-pin 的 notepad 内容加入 contexts.pinned[]
- 用户也可通过 @notepad <name> 引用特定笔记本
```

#### 4.2.6 Composer 专项 UI

**设计方案：**

```
概念: 当用户需要"从零创建一组文件"（如脚手架/新模块）时，
      提供专门的 Composer 界面，比 Agent 的单步推进更直观。

触发: 
- 命令面板 → CodePilot: New Composer
- 或在 Chat 中输入 "创建 xxx" 时自动提示切换

UI 差异（相比普通 Agent）:
- 左侧: 文件树预览（即将创建的文件结构）
- 右侧: 每个文件的内容预览
- 底部: "生成" → "全部创建" 两步确认
- 无逐步 Plan，一次性生成所有文件

实现: 本质上是对 Agent 模式的 UI 封装，后端无需特殊处理
- mode=agent, 但前端以 Composer UI 渲染
- 收集所有 patch.op=create 的文件
- 以文件树形式展示而非逐步 plan
```
