## 三、缺失能力设计方案

### 3.1 全工程代码索引（Codebase Indexing）

> 对标 Cursor 的 @Codebase 能力 — 用户无需手动上传分片，插件自动在后台完成全工程索引。

#### 3.1.1 架构决策：纯本地索引（已实现）

**关键决策：索引 100% 在插件端本地完成，不经过后端。**

理由：
1. 代码本身就在本地，上传再检索是浪费
2. IDEA PSI 是最强的代码理解引擎，比任何后端 embedding 都精准
3. 网络延迟不可接受（@codebase 需要实时响应）
4. 上传代码到后端 VectorStore 违反"后端不落库用户代码"的核心设计原则
5. Cursor 也是纯本地索引

```
插件端（纯本地，零网络依赖）
┌──────────────────────────────────────────┐
│ IndexWatcher (VFS 事件监听)               │
│  └ → 增量变更队列                         │
│ ChunkBuilder (PSI 语言感知分片)           │
│  └ → 按 class/method 边界或滑动窗口分片   │
│ LocalIndexStore (JSON 文件元数据)         │
│  └ → ~/.codePilot/index/<wsHash>/         │
│ LocalSearchEngine (内存倒排索引)          │
│  └ → TF-IDF 评分 + 符号匹配 + 路径匹配   │
│ IndexScheduler (后台调度器)               │
│  └ → 全量扫描(24h) + 增量同步(5s)        │
└──────────────────────────────────────────┘
```

#### 3.1.2 插件端设计（已实现）

```kotlin
// 新增模块：indexer
class IndexWatcher(private val project: Project) {
    // 监听 VFS 事件，增量收集变更文件
    // 排除 .gitignore / node_modules / build / .idea 等
    // 首次打开工程时执行全量索引（异步后台，不阻塞 IDE）
    // 后续文件保存时增量更新（debounce 2s）

    // 本地维护 SQLite 索引（~/.codePilot/index/<workspaceHash>/）
    // 记录每个文件的 hash、上次索引时间、chunk 列表
    // 用于快速判断哪些文件需要重新分片上传
}

class ChunkBuilder {
    // 按语言感知的方式分片：
    // - Java/Kotlin: 按 class/method 边界分片（利用 PSI）
    // - JS/TS/Python: 按 function/class 边界分片
    // - 通用: 按 ~200 行滑动窗口，50 行重叠
    // 每个 chunk 带元数据: path, language, range, symbols[], imports[]
}

class IndexScheduler {
    // 低优先级后台线程池
    // 首次全量索引：限制 CPU 使用率 ≤30%
    // 增量索引：每次批量上传 ≤50 个 chunk
    // 索引状态通知：StatusBar widget 显示进度
}
```

#### 3.1.3 搜索策略（已实现于 LocalSearchEngine）

搜索采用多策略混合排序：
1. **精确词条匹配** — TF-IDF 加权的倒排索引查询
2. **前缀/部分匹配** — 对符号名做前缀模糊匹配
3. **路径匹配加分** — 查询词命中文件路径时给予强权重
4. **语言过滤** — 可选按语言（java/kotlin/ts 等）过滤结果

本地搜索不依赖任何 embedding 模型或向量数据库。
对于需要语义理解的搜索，由 Agent 的 ContextBudgeter 在 LLM 调用时
把 @codebase 搜索结果作为 contexts.refs 注入即可。

> 对标 Cursor 的 @file / @folder / @code / @web / @docs / @git / @codebase 引用。

#### 3.2.1 设计

在 InputBar 的输入框中，当用户键入 `@` 时弹出引用选择器：

| 引用类型 | 触发 | 行为 | 实现位置 |
|---|---|---|---|
| `@file` | `@` + 文件名模糊匹配 | 把文件内容/outline 加入 contexts.refs | 插件端 |
| `@folder` | `@` + 目录名 | 把目录下所有文件的 outline 加入 refs | 插件端 |
| `@symbol` | `@` + 类/方法名 | PSI 查找 + 定义/引用代码加入 refs | 插件端 |

| `@codebase` | `@codebase` + 描述 | 全工程语义搜索（依赖 §3.1） | 后端 |

### 3.4 项目规则文件支持（.codepilot/rules）

> 对标 Cursor 的 .cursorrules 功能。

#### 3.4.1 设计

```
项目根目录下的 .codepilot/rules/ 文件夹中的 .md 文件
将被自动读取并作为 system prompt 的一部分注入每次请求。

优先级: project rules > user Skill (global) > system Skill

文件结构:
<projectRoot>/.codepilot/
├── rules/
│   ├── coding-style.md      # 编码风格规范
│   ├── architecture.md      # 架构约定
│   ├── testing.md           # 测试规范
│   └── security.md          # 安全要求

### 3.7 Shadow Workspace / Background Validation

> 对标 Cursor 的后台编译验证 — 在应用 Patch 前先在副本里验证编译通过。

#### 3.7.1 设计

```

2. 创建 Shadow Workspace（利用 IDEA 的 Light Virtual File 或临时目录）

5. 验证通过 → 允许应用到真实工作区
6. 验证失败 → 回传错误给 Agent 的 selfCheck → 触发 repair

### 3.8 Bug Finder（后台自动扫描）

> 对标 Cursor 的 Bug Finder 功能 — 后台分析当前文件/项目的潜在 Bug。

#### 3.8.1 设计

```
触发方式:
- 手动: 右键 → CodePilot → Find Bugs
- 自动: 文件保存后 debounce 30s 后台扫描（可配置开关）
- 全局: Settings → CodePilot → Background Analysis 开关

实现:
1. 利用 IDEA Inspections API 收集当前警告/错误
2. 把 diagnostics + 相关代码上下文发送给 LLM
3. LLM 分析并返回结构化的 Bug 报告:

   - location: file + line range
   - description: 问题描述

   - confidence: 置信度
4. 结果以 Inlay Hint / Gutter Icon 形式展示在编辑器中
5. 点击可直接触发 Agent 模式修复

接口:
POST /v1/actions/bug-scan
  body: { sessionId, files: [{ path, content, diagnostics[] }] }
  resp: SSE stream of { bugs: [{ severity, location, description, suggestion }] }
```

## 四、不足能力改进方案

### 4.1 代码补全质量提升

**现状问题**: 当前 InlineCompletionProvider 只做单次请求、无 FIM 优化、无质量排序。

**改进方案**:
1. **FIM (Fill-in-the-Middle)**: 后端 prompt 改为 FIM 格式 (`<PRE>prefix<SUF>suffix<MID>`)
2. **多候选排序**: 一次请求返回 3-5 个候选，按概率排序
3. **补全缓存**: 相同 prefix 缓存结果，减少请求
4. **触发优化**: 增加语义感知的触发条件（如完成 `=` 后、函数参数内等）
5. **多行补全**: 支持跨行补全（如完整的 if-else、函数体）
6. **取消管理**: 用户继续输入时取消上一次请求
7. **本地 Tokenizer**: 使用 jtokkit 在客户端估算上下文大小

### 4.2 PlanPanel / TaskLedger 前端完善

**现状问题**: PlanPanel.tsx 仅 2.3KB，功能较简。

**改进方案**:
1. 步骤树支持展开/折叠
2. 每步显示状态徽章（pending/running/success/failed）+ 风险标签
3. 支持拖拽排序
4. 支持勾选/跳过/编辑步骤
5. TaskLedger 面板：显示 goal + subtasks 卡片 + cursor 高亮 + notes
6. "继续" / "回答" / "重新规划" 快捷按钮

### 4.3 Diff 预览增强（Partial Accept）

**现状问题**: PatchApplier 只支持全量 Accept/Reject。

**改进方案**:
1. 使用 IDEA 的 `MergeRequest` 支持三方 Diff
2. 每个 hunk 独立的 Accept/Reject 按钮
3. 编辑器内 Gutter 显示 Accept/Reject 图标
4. 支持手动编辑 AI 建议的内容后再 Accept

### 4.4 上下文压缩策略完善

**现状问题**: ContextBudgeter 框架已有，但裁剪算法待实现。

**改进方案**:
1. 实现完整的分层裁剪：MUST-KEEP → 可降精度 → 可折叠 → 触发模型压缩
2. 客户端 `SessionService.prepareRequest()` 完整实现预算装配
3. 双向记忆：持久化 `summaryForNextTurn` + `hintsForContext.pin/unpin`
4. Token 估算一致性：客户端和服务端统一使用 jtokkit/tokenizer

### 4.5 断点恢复完善

**现状问题**: 基本框架有，但 checkpoint 写入/恢复流程不完整。

**改进方案**:
1. 每个 Agent 步骤完成后自动写 `checkpoints/{n}.json`
2. 恢复时读取最后 checkpoint + completedToolCalls 列表
3. 幂等校验：按 toolCallId 判断是否跳过已完成步骤
4. UI：会话列表中对有 checkpoint 的会话显示"可恢复"标记

### 4.6 NeedsInput 交互完善

**现状问题**: NeedsInputCard 基础实现，缺选项卡片 UI。

**改进方案**:
1. 渲染选项为可点击 Chip（带 impact 徽章 + pros/cons tooltip）
2. 支持"编号+答案"解析（`1: b; 2: a; 3: 600`）
3. 支持完全自由输入模式
4. 已回答的问题记录到 taskLedger.notes，防止重复追问

### 4.7 RAG 自动索引增强

**现状问题**: 需要手动上传分片，无自动触发。

**改进方案**:
1. 与 §3.1 全工程索引集成
2. Agent 模式首轮自动检测是否需要 RAG 索引
3. 支持增量索引更新（只上传变更的 chunk）
4. 索引状态可视化（StatusBar widget）

### 4.8 可观测性增强

**现状问题**: MetricsService 仅基本计数器。

**改进方案**:
1. 集成 OpenTelemetry：traceId 贯穿插件→网关→Service→LLM
2. 增加 tokens_in/out、tool_calls、latency_p95、agent_steps、repair_rate 指标
3. 插件端开发者工具面板：显示最近请求的 tracing 时间线
4. 日志脱敏强化
