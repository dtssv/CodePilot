## 3. GraphEngine核心（StateGraph编排）

### 3.1 GraphEngineService

核心服务，使用Spring AI Alibaba的StateGraph API：

```java
@Service
public class GraphEngineService {
    // 启动时编译图，每次请求invoke
    public Flux<ServerSentEvent<String>> run(ConversationRunRequest req, String userId) {
        var graph = "deep-research".equals(template) 
            ? buildDeepResearchGraph() : buildGraph();
        var initialState = intakeAction.buildInitialState(req, userId);
        // 使用Sinks.Many实时推送SSE事件
        Sinks.Many<ServerSentEvent<String>> liveSink = ...;
        // 在专用scheduler上执行图（避免阻塞ForkJoinPool）
        graphScheduler.schedule(() -> {
            GraphSseHelper.setLiveSink(liveSink, sse);
            graph.invoke(initialState.data());
        });
        return liveSink.asFlux();
    }
}
```

### 3.2 图拓扑（标准模式）

```
节点: intake → planning → preCheck → generate → applyPatch → verify → repair
      → gather → reenter → commit → askUser → finalize

边:
  START → intake
  intake → planning (fresh) / 目标节点 (resume)
  planning → preCheck / gather / finalize
  preCheck → generate / gather / askUser
  generate → applyPatch / gather / askUser / commit
  applyPatch → verify / repair / commit
  verify → commit / repair / askUser
  repair → applyPatch / askUser / commit
  gather → reenter
  reenter → planning / preCheck / generate / repair
  commit → preCheck (下一phase) / finalize
  askUser → 任意目标节点 / END
  finalize → END
```

### 3.3 图拓扑（Deep Research模式）

```
节点: intake → planning → generate → gather → reenter → searchEvaluate
      → commit → synthesize → askUser → finalize

边:
  START → intake
  intake → planning / 目标节点 (resume)
  planning → generate / gather / finalize
  generate → gather / askUser
  gather → reenter
  reenter → searchEvaluate (固定)
  searchEvaluate → commit (sufficient) / generate (insufficient) / askUser
  commit → generate (more phases) / synthesize (all done)
  synthesize → finalize
  finalize → END
```

## 4. 各Action职责与emit事件

### 4.1 IntakeAction

- 构建初始Graph状态（input/mode/modelId/userId/projectMeta等）
- Resume时：从checkpoint恢复状态，合并用户答案
- 路由：fresh→planning，resume→目标节点
- emit: graph_transition

### 4.2 PlanningAction

- 调用LLM生成双层Plan（UserPlan + ExecutionPhases）
- **skipPlan机制**：LLM返回`skipPlan=true`时跳过详细规划，直接进入执行
- 解析LLM返回的JSON：`{userPlan, phases[], infoRequests[], skipPlan, agentThinking, agentContent}`
- emit:
  - `agent_thinking`（思考状态）
  - `user_plan`（当skipPlan=false时）
  - `graph_plan`（内部phases）
  - `graph_transition`

### 4.3 PreCheckAction

- 执行前置检查（权限/依赖/上下文完整性）
- emit: `graph_transition`
- 路由：通过→generate，缺信息→gather，需确认→askUser

### 4.4 GenerateAction

- 调用LLM生成代码变更（patches/toolCalls/textOutput）
- **agentContent/agentThinking/agentWriting分离**：
  - `agentThinking`：简要思考（"我已分析你的需求"）
  - `agentContent`：结构化用户内容（含`<plan>`/`<file>`/`<run>` XML标签）
  - `agentWriting`：写操作预览（文件路径+行数）
- **反循环注入**：gather已执行时，强制LLM产出patches而非infoRequests
- 解析LLM返回：`{patches[], infoRequests[], askUser, textOutput, agentThinking, agentContent}`
- emit:
  - `agent_thinking` / `agent_writing` / `agent_reading`（语义层）
  - `delta`（agentContent或textOutput）
  - `graph_transition`

### 4.5 ApplyPatchAction

- **批量Patch机制**：多个Patch合并为单个tool_call减少N×RTT
- 过滤无效Patch（path为空/unknown，内容为空）
- 可选ShadowWorkspace预验证
- 发tool_call SSE→等待ToolResultBus回传结果
- 批量失败时回退到逐个Patch执行
- emit:
  - `tool_call`（批量patches）
  - `agent_writing`（写操作语义事件）
  - `graph_transition`

### 4.6 VerifyAction

- 执行编译/lint验证（可选ShadowWorkspace）
- emit: `graph_verify`, `agent_running`
- 路由：通过→commit，失败→repair，不确定→askUser

### 4.7 RepairAction

- LLM分析失败原因，生成修复Patch
- emit: `graph_repair_plan`, `agent_thinking`
- 路由：修复→applyPatch，无法修复→askUser，部分成功→commit

### 4.8 GatherAction

- 执行信息收集（fs.read/fs.list/fs.search等）
- 发graph_info_request SSE→等待客户端通过ToolResultBus回传
- emit: `graph_info_request`, `agent_reading`
- 路由：收集完成→reenter

### 4.9 ReenterAction

- 将GatherAction收集的信息合并到gatheredInfo状态
- 路由：回到发起gather的原始节点（planning/generate/repair等）

### 4.10 CommitAction

- 标记当前phase完成
- emit: `graph_phase_done`, `user_plan_progress`
- 路由：还有未完成phase→preCheck，全部完成→finalize

### 4.11 AskUserAction

- 抛出GraphInterruptException中断图执行
- 保存checkpoint到Redis（含continuationToken+nextNode）
- emit: `needs_input`
- 路由：用户回答后resume到目标节点

### 4.12 FinalizeAction

- 发送最终结果和done事件
- emit: `delta`(最终答案), `done`(含reason+continuationToken)

### 4.13 SearchEvaluateAction（Deep Research专用）

- 评估搜索结果是否充分
- emit: `graph_transition`
- 路由：充分→commit，不充分→generate(重试)

### 4.14 SynthesizeAction（Deep Research专用）

- 汇总所有子问题的搜索结果，生成最终报告
- emit: `delta`(综合报告)

## 6. ToolResultBus（异步工具调用回传）

### 6.1 双机制架构

```
┌─ 等待线程 (ApplyPatchAction/GatherAction) ──────────┐
│  1. registerFuture(sessionId:toolCallId, future)     │
│  2. emit tool_call SSE                              │
│  3. future.get(timeout) 等待结果                     │
└─────────────────────────────────────────────────────┘
        ↕ CompletableFuture (单实例直接交付)
        ↕ Redis Pub/Sub (跨副本交付)
┌─ HTTP Handler (POST /v1/conversation/tool-result) ───┐
│  1. publish(sessionId, toolCallId, result)           │
│  2. 本实例: complete Future                          │
│  3. 跨实例: publish to Redis channel                 │
└─────────────────────────────────────────────────────┘
```

- **CompletableFuture**（主）：ConcurrentHashMap<sessionId:toolCallId, Future>
- **Redis Pub/Sub**（辅）：channel=`codepilot:tool-result:{sessionId}`，跨副本时桥接到Future

### 6.2 超时与清理

- 默认超时120s（ApplyPatch），60s（Gather）
- 超时后Future自动complete with timeout error
- 请求完成后unregister Future

## 7. GraphSseHelper（实时事件推送）

- **ThreadLocal liveSink**：当前线程的SSE sink
- **Session-based Map**：跨线程访问（M1修复，异步节点在不同线程）
- `emitEvent(state, eventName, data)` → 从ThreadLocal或SessionMap获取sink → push事件
- 请求结束后清理ThreadLocal和SessionMap

## 10. ContextBudgeter（服务端二次裁剪）

6阶段递进裁剪策略：

| 阶段 | 策略 | 是否可逆 |
|---|---|---|
| 1 | MUST-KEEP（不裁）：goal/cursor/pinned/completedToolCalls/projectRules | 否 |
| 2 | 丢弃最早recent消息（保留至少2条） | 是 |

| 4 | 折叠recent→localDigest | 否 |

## 11. SkillRouter（按需加载领域Skill）

### 11.1 双域分治

- **system域**：驻后端，永不下发到客户端。商城仅返回安全摘要。
- **user域**：完全位于客户端本地，每次请求上行userSkills[]，后端临时注入用完即丢。

### 11.2 路由算法

```
1) 候选 = 后端system域候选(triggers匹配) + 请求体userSkill候选
2) 显式白名单过滤 (skills.requested[])
3) 黑名单过滤 (skills.disabled[])
4) 安全约束 (user域禁止override system, 权限校验)
5) 冲突解决 (project>global, system>user)
6) 限额 (lang.primary=1, lang.aux=2, scenario=2)
7) 排序 (source>priority>scope>version)
```

## 13. 安全设计

| 维度 | 措施 |
|---|---|
| 模型安全 | 上行二次脱敏；Prompt Injection检测；越权工具调用拦截 |
| 系统提示词机密性 | SystemPromptLeakFilter（前置+后置双拦截） |
| 用户隐私 | 后端不落库会话/代码；user Skill仅临时注入 |
| 传输安全 | HTTPS + HSTS + HMAC签名 + 时间戳 + nonce |
| 输出安全 | 模型输出做代码注入/凭证/PII/系统提示词外泄扫描 |
| Skill安全 | system由后端自管；user仅本地；运行时隔离 |

### 13.1 SystemPromptLeakFilter

**前置**：网关LeakIntentDetector检测窃取意图（关键词+小模型），命中直接返回拒绝。

**后置**：响应流必经SystemPromptLeakFilter，对delta/patches做关键词匹配+大段相似度检测（Rabin-Karp 30-gram），命中替换为标准拒绝并审计。
