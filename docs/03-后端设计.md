## 3. 模型层（model-service）

### 3.1 ChatClient 工厂

使用 Spring AI 的多 provider 支持，通过 `ChatClient.Builder` 构建：

```java
@Configuration
public class ChatClientFactory {
  @Bean
  ChatClient defaultClient(OpenAiChatModel openai,
                           OllamaChatModel ollama,
                           PromptRegistry prompts) {
    return ChatClient.builder(openai)
        .defaultSystem(prompts.system("default"))
        .defaultAdvisors(
            new SimpleLoggerAdvisor(),
            new SafeguardAdvisor(),               // 自定义：脱敏/审核
            new MessageChatMemoryAdvisor(memory)  // 短期记忆，不落库
        )
        .build();
  }
}
```

### 3.2 自定义服务商

- 注册接口：`POST /v1/models`，body：
  ```json
  {
    "name": "my-azure-gpt4o",
    "protocol": "openai",
    "baseUrl": "https://xxx.openai.azure.com/v1",
    "apiKey": "encrypted...",
    "model": "gpt-4o",
    "headers": {"api-version": "2024-08-01"},
    "timeoutMs": 60000
  }
  ```
- 后端用 `OpenAiApi.builder().baseUrl(...).apiKey(...).build()` 构造一个临时 `OpenAiChatModel`，缓存到 Caffeine。
- ApiKey 在数据库以 KMS 加密存储；返回前端永远只回 `***末4位`。

### 3.3 内置模型

由运维通过配置中心下发清单（含模型能力位：`tools`, `vision`, `maxTokens`, `pricingHint`），插件端拉取后展示。

## 4. Prompt 仓库（prompt-registry）

- 存储：YAML 文件 + DB 表（DB 优先），按 `taskType / language / version` 索引。
- 模板引擎：StringTemplate / Mustache / Spring AI `PromptTemplate`。
- 灰度：基于 `userId hash` 命中实验组。
- 关键模板见 `04-Prompt模板.md`。

## 5. 工具协议（统一 Schema）

所有工具（无论是插件端 FileOps、Shell、还是 MCP Server 提供的）统一以 OpenAI function-calling 风格描述：

```json
{
  "name": "fs.replace",
  "description": "Replace text in a file",
  "parameters": {
    "type": "object",
    "properties": {
      "path":   {"type": "string"},
      "search": {"type": "string"},
      "replace":{"type": "string"},
      "regex":  {"type": "boolean", "default": false},
      "range":  {"type": "object",
                 "properties":{"startLine":{"type":"integer"},"endLine":{"type":"integer"}}}
    },
    "required": ["path","search","replace"]
  },
  "executor": "client"   // client=插件端执行；server=后端执行；mcp=mcp 服务执行
}
```

`ToolRouter` 根据 `executor` 选择执行端：
- `client`：以 ToolCall 通过 SSE 下发给插件，等待 ToolResult 通过 `POST /v1/conversation/tool-result` 回传。
- `server`：后端直接执行（如 RAG 检索）。
- `mcp`：后端经 MCP Hub 代理到对应 MCP Server。

## 6. Agent 循环（conversation-service，多轮多次往返）

设计原则：
- **Chat 与 Agent 共用一个端点 `/v1/conversation/run`**，由 `mode` 字段路由。
- **一次 run 推进一段，不是做完整件事**。复杂任务通常由插件端发起 N 次 `/conversation/run`：每次只完成一小段（如"先读规范"或"实现一个子任务"），到达自然停顿即 `done`。复杂上下文（plan / ledger / completedToolCalls / digest）由插件端落本地，下次随请求带回来。
- **PromptOrchestrator 多段式 system**：每次 run 拼装 base + agent + tools + (按需)compact/replan/userEdits/selfCheck/needsInput/riskNotice/styleLearning/patching/languageProfiles/repairLoop/resume/delivery + guard，让规划 / 压缩 / 自检 / 风险预告 / 修复 等行为都在**同一次模型调用**里完成。
- **不暴露独立的 Plan / Compact / SelfCheck 接口**：所有这些都是 Prompt 行为，体现为 SSE 事件 `plan / plan_delta / digest / self_check / risk_notice / needs_input / task_ledger`。
- **每"轮" = 1 次模型调用 + ≤1 次工具执行 + 工具结果回传**。整个 run 是若干轮串在同一 SSE 通道里。
- **`done.reason` 描述本次 run 为什么停**：`final` 任务完成 / `subtask_done` 一个子任务做完 / `awaiting_user_input` 模型问问题 / `paused` 用户主动暂停 / `max_steps` 超步数 / `stopped` 主动中断 / `failed` 致命错误。
- **mode=chat** 时禁用 plan/toolCall/selfCheck/needsInput/riskNotice/taskLedger，只允许 `digest + delta + usage`。

伪代码：

```java
public Flux<AgentEvent> run(AgentRequest req) {
  return Flux.create(sink -> {
    State state = stateLoader.load(req);             // 含历史 messages、已完成 steps、上一轮 plan
    PolicyDecision pd = orchestrator.decide(state, req.policy()); // 是否需要 compact / replan 提示

    while (state.steps() < MAX_STEPS) {
      // 1) 单次模型调用（同时承担：可选 digest + plan/planDelta + toolCall|final）
      Prompt p = orchestrator.assemble(state, pd);   // 多段 system 串联：base+agent+tools+(compact?)+(replanHint?)
      AgentTurn turn = chatClient.callStrict(p, AgentTurn.class);  // BeanOutputConverter
      if (turn.digest() != null) {
        sink.next(new CompactedEvent(turn.digest()));
        state.applyDigest(turn.digest());
      }
      if (turn.plan() != null)      sink.next(new PlanEvent(turn.plan()));
      if (turn.planDelta() != null) sink.next(new PlanDeltaEvent(turn.planDelta()));

      if (turn.isFinal()) { sink.next(new DoneEvent(turn.answer(), "final")); break; }

      ToolCall call = turn.toolCall();
      sink.next(new ToolCallEvent(call));            // 客户端执行 → 异步回 result
      ToolResult r = router.awaitClientResult(req.sessionId(), call.id());

      state.append(turn, r);
      checkpointStore.save(req.sessionId(), state.snapshot()); // 仅 Redis 短 TTL

      // 失败修复：不另起一次"修复 prompt"，只是让下一轮模型在 planDelta 中把失败步骤标记为 failed 并补充修复步骤
      pd = orchestrator.decide(state, req.policy());
    }
    sink.complete();
  });
}
```

输出契约（一次模型调用的回复 JSON，由 `BeanOutputConverter` 严格校验）：

```json
{
  "digest": null,                          // 可选：命中压缩条件时存在；结构见 §6.2
  "plan":     null,                        // 首轮或 Replan 时全量 Plan
  "planDelta":null,                        // 后续轮：仅修改的步骤
  "thought":  "≤200 字的简要思考",         // 仅日志，不下发用户
  "toolCall": {"id":"tc-1","name":"fs.read","args":{...},"riskLevel":"low"},
  "final":    null                         // 当 toolCall 为空时，可填 {"answer":"...markdown..."}
}
```

约束：
- `MAX_STEPS` 默认 25；连续 3 轮 toolCall 失败会自动让 `orchestrator.decide` 注入 `replanHint=true`，下一轮模型必须给出 `plan`（全量 Replan）。
- 一次回复中 `toolCall` 与 `final` 互斥；`plan` 与 `planDelta` 互斥。
- 任何写/删/移/shell 工具调用必须带 `riskLevel`，插件端据此处理确认。

## 6.1 PromptOrchestrator（多段式 system 拼装）

职责：把"一次调用"所需的多种行为通过**多段式 system**串联，避免被拆成多个 API。

输入：`state, policy, requestCompact, replanHint, planEdits, contexts, toolsSchema`

拼装顺序（由上到下追加为 system）：

1. `base.system` —— 通用人设与语言/风格规范。
2. `agent.system` —— Agent 输出契约（必须输出 plan/planDelta + 至多一个 toolCall）。
3. `tools.schema` —— 当前可用工具的 JSON Schema（来自插件端工具清单 + 已启用 MCP）。
4. `compact.system`（条件命中时追加）—— "在本次回复里先输出 digest，再继续 plan/toolCall"。
5. `replan.system`（命中 replanHint 时追加）—— "本轮必须输出全量 plan，重置剩余步骤"。
6. `userEdits.note`（plan 面板里有未消费的用户编辑）—— "用户对计划做了如下修改，请在 planDelta 中反映"。
7. `guard.system` —— Prompt 注入防御 + 数据-指令隔离。

> 这样**一次模型调用**就能同时完成"先做计划 → 必要时压缩上下文 → 给出本步操作"，不再分开发多个 prompt。

## 6.2 上下文压缩（在同一 Prompt 内完成）

数据：`digest` 字段写在模型回复 JSON 中，不再单独走接口。

`Digest` 结构：
```json
{
  "boundarySeq": 123,                       // 该 seq 之前的消息可被折叠
  "goal": "...",
  "decisions": ["选用 Redis SETNX 实现幂等"],
  "openQuestions": ["是否兼容老版本客户端?"],
  "keyFiles": [{"path":"src/.../OrderService.java","why":"主要被改造的服务"}],
  "completedSteps": [{"id":"s1","summary":"..."}],
  "pendingHints": ["下一步应在 createOrder 入口加幂等键校验"]
}
```

触发条件（在 `PromptOrchestrator.decide` 内判断）：
- 客户端显式 `policy.requestCompact=true`；或
- 服务端 token 估算 > 模型上下文 70%；或
- Agent 步骤 > 12 / 消息条数 > 80。

并发与幂等：
- 同一 sessionId 内只允许同时存在一轮 `/conversation/run`（网关层用 Redis 锁 `lock:agent:{sid}` 保证）。
- digest 仅在 Redis 短期缓存（TTL 24h）；权威副本由插件端落 `summaries/` 文件。
- 安全：digest 不得复制大段代码，仅引用 `path:lineRange`；`SafeguardAdvisor` 上下行二次脱敏。

## 6.3 ContextBudgeter（服务端二次裁剪 + 层级注入）

职责：在 ChatClient 发起之前，把插件端送来的分层请求**二次裁剪**到本次模型调用的"预算"，并按层级映射到 system/user/tool 消息序列。

输入：
- `request.contexts = { pinned[], recent[], refs[] }`
- `request.taskLedger` / `lastPlanDigest` / `lastAssistantTurnSummary` / `sessionDigest`
- `request.completedToolCallsTail` / `earlierToolCallsCount`
- `policy.contextBudgetTokens` / `policy.keepRecentMessages`

裁剪策略（按层优先级）：
1. MUST-KEEP（不裁）：`goal`, `cursor 分支`, `已确认决策`, `pinned`, `completedToolCallsTail`。
2. 可降精度：`recent` → 按 FIFO 丢；`refs` 仅保留 path+range+sha1，不展开内容。
3. 可折叠：多条 `recent` → `miniDigest`（本地数学化摘要，非模型调用）。
4. 触发模型压缩：预算仍不够 → 把 `prompt.agent.compact`（§9.2）加入 system，让模型**本轮先输出 digest** 再继续；并把 `recent` 全部降为 summary 行。
5. 最终编排映射：

```
system   = base + agent + contextBudget + (compact?) + (replan?) + (userEdits?)
                + styleLearning + patching + languageProfiles
                + (selfCheck on turn>=2) + (resume?) + (delivery?) + tools.schema + guard
user[0]  = {
  "input": request.input,
  "intent": request.intent,
  "answers": request.answers?,
  "taskLedger": 极简,
  "lastPlanDigest": 极简,
  "lastAssistantTurnSummary": ≤400 tokens,
  "sessionDigest": request.sessionDigest?,
  "contextRefs": request.contexts.refs,
  "completedToolCallsTail": request.completedToolCallsTail,
  "earlierToolCallsCount": n
}
user[1..] = request.contexts.recent (原文或摘要)
tool[]    = request.completedToolCallsTail 映射的 tool 结果（必要时只带摘要）
```

补充能力：
- **TokenMeter**：统一 tokenizer（与客户端 jtokkit 一致），确保双端口径一致。
- **LazyRefResolver**：当模型在工具调用中访问到某个 ref（`fs.read path range`）时，按 sha1 校对客户端现有版本再返回。
- **RedactFilter**：二次脱敏（邮箱/手机号/身份证/AK-SK）。
- **EnvelopeValidator**：模型回复 JSON 严格校验，失败一次让模型"仅输出修正后的 JSON"。

审计：
- 不落库用户内容，但会记录一条 **PromptBudgetEvent**（只含 tokens/层大小/是否触发 compact），用于 SLA 与成本分析。

## 6.4 NeedsInput 处理（needs_input 事件 + answers）

- 后端不做答案解析（解析由插件端完成，遵循 §05-3.2.1）；后端只做：
  - 校验 `answers[*].questionId` 在上一次 SSE 的 `needs_input` 集合内（如果能从 Redis 取到）；
  - 校验 `optionId` 属于对应问题；
  - 如果 `answers[*].freeform` 含危险关键词（如密钥模式），走 `SafeguardAdvisor` 先脱敏再入 prompt；
- 当 `answers` 非空且 `intent="answer"` 时：
  - 在 system 头部追加 `[ANSWERED_QUESTIONS]` 明确罗列"问题→答案"映射，禁止模型再次追问；
  - 如 `answers` 里出现完全无法对齐的 `freeform`，在 user 消息中以 `[USER_FREEFORM_INPUT]` 传入，允许模型重新规划；
- 当模型在响应中再次发出 `needsInput`：
  - `PromptOrchestrator` 校验"不得与上一次相同 `questionId` 重复"；若重复 → 后端直接拦截并要求模型重写（一次自动纠错）。

## 6.5 SkillRouter（按需加载领域 Skill）

职责：在 `PromptOrchestrator` 拼装 system 之前，根据本轮上下文挑选并合并要激活的 Skill 集合，与骨架 workflow 共同组装出最终 system。

### 输入信号

- `mode` / `action` / `intent`；
- `language` / `framework`（来自 ContextBudgeter 推断的 StyleProfile，或客户端提供）；
- `filePath` / `fileGlob`（来自 contexts.refs/pinned）；
- `priorTurnFailed`（来自最近一次 selfCheck.ok）；
- `approachingDelivery`（plan 接近完成 / cursor=最后一个子任务）；
- `tenant / workspace / user`（多租户上下文）；
- 请求体显式开关：`skills.requested[]`（白名单，钉版本）/ `skills.disabled[]`（黑名单）。

### Skill 注册表（SkillRegistry）

后端 SkillRegistry **只持有 system 域**，并保证以下"系统 Skill 不可下发"红线：

- 来源（system 域）：
  - `builtin/`：随后端发版内置；
  - `marketplace.system/`：平台审核、签名后内置进商城的 system Skill。
- 索引：`(id, version)` 主键；通过 `triggers` 建立倒排（按 language/action/fileGlob 等）。
- 缓存：Caffeine LRU；变更通过 PubSub 失效。
- **system Skill 永不下发到客户端**：
  - 任何 API 都**不**返回 `systemPrompt` / `examples` / `tests` 等可能含提示词正文的字段；
  - 商城详情接口仅返回"安全摘要"（id/version/title/triggersBrief/permissions/audit.tokensEstimate/changelog 链接）；
  - 客户端不存放、不缓存 system Skill 原文；浏览/筛选都基于安全摘要；
  - 所有 system Skill 的拼装与使用仅发生在后端 `PromptOrchestrator` 内部，使用后随响应释放，不写日志原文。
- 热更新：当 `marketplace.system` 推送 / `builtin` 通过配置中心更新时，PubSub 失效缓存，后续请求立即用新版本（详见 §9.x 客户端无感知，因为它从不持有 system Skill 原文）。
- **不存放任何 user 域 Skill**。

user 域 Skill / MCP **完全位于客户端本地**：

- 物理位置：`~/.codePilot/skills/`、`~/.codePilot/mcps/`（路径可改，详见 Prompt §A.1.2）。
- 每次 `/conversation/run` 由插件端把当前命中的 user Skill 通过请求体上行（接口 §3.1.2 `userSkills`）；后端仅在本次请求中临时注入到 system 末尾、用完即丢弃，**不写日志原文、不入数据库、不进缓存**（仅可缓存 sha256/version 用于幂等审计）。
- 校验：
  - 必须带 `source: user`；否则拒绝；
  - `permissions.tools` 必须是当前 `tools[]` 白名单的子集；
  - 禁止 `merge: override` 任何 `source=system` 的 Skill；
  - 总 token 与单 Skill token 上限校验（见 Prompt §B）。

### 路由算法

```java
List<Skill> activate(SkillContext ctx) {
  // 1) 候选 = 后端 system 域候选 + 请求体里 user Skill 候选
  Set<Skill> candidates = registry.matchByTriggers(ctx);                     // system only
  candidates.addAll(ctx.userSkills().filter(s -> s.matches(ctx.signals())));  // request-scoped only

  // 2) 显式白名单
  if (ctx.requested().nonEmpty()) {
    candidates = filterByRequested(candidates, ctx.requested()); // 钉 id@version
  }
  // 3) 黑名单
  candidates.removeIf(s -> ctx.disabled().contains(s.id()));

  // 4) 安全约束（user 域）
  candidates = enforceUserConstraints(candidates, ctx);
    // - source=user 的 merge 必须是 append 或 wrap，不能 override
    // - permissions.tools ⊆ current tool whitelist
    // - 不允许覆盖 envelope / selfCheck / needsInput / guard
    // - scope 必须是 project 或 global；scope=system 视为违规（system 由后端自管）
    // - 相同 id 同时存在 project 与 global：按 project > global 仅保留一条
    // - project 作用域的 Skill 必须携带本次会话 request.projectRootHash，与客户端上行一致

  // 5) 冲突
  candidates = resolveConflicts(candidates);

  // 6) 限额
  candidates = capByCategory(candidates, Map.of(
      "lang.primary", 1,
      "lang.aux",     2,    // SQL/cache/NoSQL/messaging/IaC/shell 共享
      "scenario",     2));

  // 7) 排序：source(system>user) > priority desc > scope(project>global>system) > version desc
  return candidates.stream().sorted(Skill.ORDER).toList();
}
```

### 与 PromptOrchestrator 的协作

```
PromptOrchestrator.build(req)
  ├── ContextBudgeter.shape(req)        // 见 §6.3
  ├── skeleton = composeSkeleton(req)   // §0 / §9.1 / §9.5 / §9.6 / §9.6a / §9.7 / §9.8 / §10
  ├── skills   = SkillRouter.activate(ctx)
  ├── system   = mergeSkills(skeleton, skills)   // append/override/wrap，see §A.3
  ├── userMsg  = composeUser(req)       // input/answers/refs/...
  └── return Prompt(system, userMsg, tools)
```

### 校验与拒绝

- Skill 声明的 `permissions.tools` 必须是当前 `tools[]` 白名单的子集；不满足则不激活并审计 `skill_permission_denied`。
- Skill `dependsOn` 缺失：自动尝试激活其依赖；仍缺失则不激活并审计 `skill_missing_dependency`。
- `conflicts` 互斥：返回 `error.skill_conflict`；客户端可在 `skills.requested / disabled` 里调整。
- token 超预算：按 `priority` 从低到高丢弃，直到落入 `policy.contextBudgetTokens`；丢弃事件入 `PromptBudgetEvent`。

### 商城与签名

- 复用 `mcp-hub`（§10）数据模型，新增 `mcp_package.type='skill'`。
- 安装/升级仅记录元数据；Skill 文本由插件端缓存，并允许后端按租户层级覆盖。
- 强制安全：第三方 Skill 不允许在 `systemPrompt` 内出现工具白名单外的工具名；不允许携带运行时代码（仅声明性 prompt + yaml）。

### 可观测性

- 每次请求记录：`activatedSkills=[id@version,…]`、`tokensBySkill={id:n}`、`droppedDueToBudget=[…]`、`mergeOps=[append|override|wrap]`。
- A/B：`skills.requested` 可被实验平台覆写，进行灰度。

## 7. 断点续跑（resume）

- 入参：`sessionId, sessionDigest, lastCheckpoint, plan?`。
- 后端不持久化 checkpoint，仅在 Redis 缓存 24h（key=`agent:ck:{sid}`）。
- 续跑时插件上传本地 checkpoint（信任源是用户本地），后端按摘要 + 已完成 step 列表继续推理。
- 副作用幂等：每个工具调用插件端持久化 `toolCallId`，恢复时若 `toolCallId` 已存在结果则跳过执行。

## 8. RAG（rag-service）

- 触发：Agent 中的 `rag.search` 工具，或 Chat 模式自动 Advisor。
- 流程：插件端按需上传"分片+元数据"（不上传整工程）→ 后端计算 embedding → 存 PGVector，TTL 24h，key 绑定 `sessionId`。
- 检索：`top-k=8, mmr=0.7`，再交给 `QuestionAnswerAdvisor`。
- 隐私：embedding 向量与原文均在 `sessionId` 维度隔离；TTL 到期物理删除。

## 9. A2A 多智能体编排

适合复杂任务（如"评审 + 重构 + 生成测试 + 跑测试 + 修复"）：

```
[Coordinator] ──► [Reviewer] ──► [Refactorer] ──► [TestWriter] ──► [Runner] ──► [Fixer]
                                          ▲                          │
                                          └──────────────────────────┘
```

实现：每个 Agent 是一个独立 ChatClient + 工具集 + System Prompt，使用 Spring AI 的 `ToolCallback` 互调。Coordinator 负责选择下一个 Agent 与终止条件。

## 11. 安全设计

| 维度 | 措施 |
|---|---|
| 模型安全 | 上行二次脱敏；Prompt Injection 检测（关键词 + LLM 审核）；越权工具调用拦截（白名单） |
| 系统提示词机密性 | **SystemPromptLeakFilter（响应流必经）**：对模型输出做后置扫描（详见 §11.1），命中"系统/Skill 提示词外泄"时整轮替换为标准拒绝回复并审计 `system_leak_blocked`；用户提问命中"窃取意图"时（如 ignore previous instructions / dump system prompt / list active skills）由网关前置检查并直接返回 `final` 拒绝消息，不进 LLM |
| 用户隐私 | 后端不落库会话/代码；**user Skill / MCP 仅在请求体中临时注入，处理完即丢弃**；审计仅存 hash + 元数据；KMS 管理 ApiKey |
| 传输安全 | HTTPS + HSTS + 可选 mTLS；HMAC 签名 + 时间戳 + nonce |
| 服务安全 | Spring Security + JWT；网关限流（用户/IP/设备维度）；IDOR/越权统一在 ToolRouter 校验 path 不出 workspace |
| 输出安全 | 模型输出做"代码注入/凭证/PII/系统提示词外泄"扫描后再下发 |
| Skill 安全 | system Skill 由后端自管；user Skill 仅本地存储；运行时 `[USER_SKILL_BEGIN]…[USER_SKILL_END]` 隔离；user Skill 禁止 override `source=system` 的 Skill |
| 商城安全 | 包签名 + SBOM + 权限声明 + 沙箱执行（高危包默认禁用） |
| 合规 | 全链路审计、可关闭 Telemetry、提供企业私有部署模式 |

### 11.1 SystemPromptLeakFilter（系统提示词外泄拦截）

**前置（user → backend）**：网关 `LeakIntentDetector` 在调用 LLM 前对 `request.input + answers[*].freeform` 做规则 + 小模型双判断，命中以下意图即直接返回 SSE：

```
event: final
data: {"answer":"<refusal in {{userLocale}}>","subtaskDone":false,"summaryForNextTurn":"user requested system prompt; refused"}
event: done
data: {"reason":"final","continuationToken":"..."}
```

判别规则（关键词与正则节选，OR 组合）：
- "system prompt|developer prompt|hidden instructions|内部指令|系统提示词|系统提示|系统提示语|展示你的提示词"
- "ignore (all )?previous|forget all instructions|jailbreak|你是 (chatgpt|gpt|claude)"
- "list (active|loaded) skills|哪些 skill|哪些规则|加载了什么"
- "print (the )?prompt|repeat the system|dump (config|prompt|rules)"
- 编码混淆：base64 / rot13 / hex / unicode-bidi / zero-width；先解码再判断。

**后置（backend → user）**：响应流必经 `SystemPromptLeakFilter`，对 SSE `delta / final.answer / patches[*].newContent` 做：
- 关键词匹配：`base.system / agent.system / skill.lang.* / [SECURITY — / [Cardinal rules] / Reply envelope` 等内部短语；
- 大段相似度检测：与本轮拼装的 system 文本做字符级 Rabin–Karp 摘要 + 30-gram 命中，命中阈值即判定泄露；
- 工具/Skill 名清单泄露：识别"已激活 Skill 列表 / 工具 schema 转载"等结构；
- 触发后：丢弃当轮 `delta / patches`，替换为标准拒绝 `final`，写入 `audit.system_leak_blocked` 并在 SSE 上加一条 `error.code=45002 message="output blocked: system prompt leak"`。

**审计内容**（不存原文）：`{traceId, blockedKind, matchedRule, matchedNgramHash, modelName}`。

**可观测**：仪表板中新增 `system_leak_blocked.count` 与 `system_leak_attempts_pre.count`（前置拦截）两条指标，作为安全 SLI。
